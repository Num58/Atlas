# PrimeAtlas 信息架构候选建议

> 日期：2026-07-20  
> 状态：**评审候选，未冻结**  
> 决策层级：产品信息架构方向建议；不等同于页面清单、路由表或开发 Spec  
> 输入依据：身份迁移为最高主轴；体能单域深扎 + 融合可见；最多 3 个活跃成长域、每域可有多个目标；v6 六 Tab 仅作体验与迁移输入

---

## 0. 决策摘要

### 当前需要决定什么

本轮只需要选出一个可继续验证的一级导航方向，判断它能否稳定承载以下主循环：

> 身份定义 → 目标与里程碑 → 专业计划 → 今日执行与反馈 → 成果证据 → 身份与目标重校准

本轮**不冻结**具体路由、页面数量、Tab 文案、模块交付版本和终局 IA。

### 建议结论

建议优先进入 Spec 验证的方向为：

> **方案 A：生命周期四 Tab——今日 / 旅程 / 成长域 / 我的**

推荐理由：它以“今日行动—长期迁移—专业深扎—个人治理”组织产品，比 v6 的“身体/训练/目标/知识”工具并列更能表达身份迁移，也能让体能专业深度和跨域融合同时可见。

但该建议仍需在 Spec 阶段通过以下验证后才能锁定：

1. “旅程”是否能让用户准确理解身份、目标和成果证据的关系；
2. “成长域”是否能同时承载体能深扎、域内多个目标和知识入口，而不退化为工具目录；
3. 今日页是否能在不制造完成压力的前提下呈现身份邻近度、计划理由和融合证据；
4. 单域、双域、三域以及每域多目标时，导航与信息密度是否稳定；
5. v6 高频能力迁移后，关键任务的到达步数是否可接受。

### 三套候选对比

| 判断维度 | 方案 A：生命周期四 Tab | 方案 B：身份主轴四 Tab | 方案 C：v6 收敛五 Tab |
|---|---|---|---|
| 一级导航 | 今日 / 旅程 / 成长域 / 我的 | 今日 / 身份 / 计划 / 我的 | 今日 / 身体 / 训练 / 成长 / 我的 |
| 身份迁移显著度 | 高 | 最高 | 中 |
| 体能专业深度可发现性 | 高 | 中 | 最高 |
| 融合可见承载力 | 高 | 中高 | 中 |
| 多域扩展稳定性 | 高 | 中高 | 中低 |
| 与 v6 心智连续性 | 中 | 低 | 高 |
| 初步建议 | **推荐进入 Spec 验证** | 作为强主轴对照组 | 作为低迁移成本对照组 |

## 1. 设计与范围边界

### 1.1 IA 设计原则

1. **身份迁移优先于功能分类**：一级导航首先回答“我正在成为谁、现在走到哪里”，而不是罗列身体、训练、目标、知识等工具。
2. **今日是执行入口，不是产品最高主轴**：今日页连接身份目标与现实行动，但不以待办、连续天数或完成百分比定义成长。
3. **成长域是有限工作空间**：最多 3 个活跃成长域；每个域内部允许多个目标。不得将“最多 3 个域”误写成“最多 3 个目标”。
4. **体能单域深扎**：早期体能需要保留身体状态、专业计划、动作学习、RPE、疼痛与安全理由等深度，不可因导航收敛而被压成一张通用卡片。
5. **融合长在主流程**：融合通过目标关联、派生内容、来源芯片、知识注入评估和今日融合证据出现，不新增独立“融合任务中心”或额外打卡体系。
6. **单域用户完整可用**：未激活域不展示、不占位、不存储无意义数据、不进入进度权重。
7. **安全信息不可被层级隐藏**：风险、否决理由、安全替代和复评条件在相关计划与执行路径直接可达。
8. **v6 是候选素材库，不是导航合同**：保留已验证的高质量交互，重新组织其入口与层级。

### 1.2 本建议不新增的范围

本建议不引入新的社区、排行榜、通用聊天主界面、独立英语课程、通用日历、任务市场、装备商城或额外 Agent 产品形态。候选结构只重新组织现有 PRD 中的身份、目标、计划、执行、体能、知识、成果、洞察、调性与数据治理能力。

### 1.3 视觉系统前置边界

IA 进入线框验证前沿用统一基础约束，不在三套候选间切换视觉语言：

- 图标：仅使用 Lucide，16 / 20 / 24px；不以 emoji 作为功能图标；
- 字体：Inter + Noto Sans SC；数据与规则编号使用 JetBrains Mono；
- 行业基调：AI 原生个人成长工具，参考 Linear 的信息克制、Notion 的层级组织和 Apple HIG 的移动可达性；
- 颜色、间距、圆角、阴影与动效继续使用项目 Design Token，不因候选方案硬编码；
- 所有候选必须支持 focus-visible、键盘/辅助技术等效入口和 prefers-reduced-motion。

## 2. 方案 A：生命周期四 Tab（推荐进入 Spec 验证）

### 2.1 一级导航

1. **今日**：今日状态、安排理由、主行动、执行反馈与今日产生的身份证据。
2. **旅程**：当前身份、目标身份、身份差距、目标/里程碑、成果证据、周期回顾与重校准。
3. **成长域**：最多 3 个活跃域的专业工作空间；早期重点承载体能深扎，并容纳域内多个目标及相关知识。
4. **我的**：画像与版本、授权、数据来源、调性、通知、同步、导出/删除和其他个人治理。

建议将全局创建或调整动作放入页面上下文，而不是增加第五个固定功能 Tab。最终具体入口由 Spec 决定。

### 2.2 信息层级示意

```text
今日
├─ 为什么今天这样安排
├─ 今日状态与安全提示
├─ 主行动 / 可选 Prime Pulse
├─ 执行与反馈
└─ 今日身份证据 / 融合证据

旅程
├─ 当前身份 → 目标身份
├─ 活跃成长域概览（≤3）
├─ 每域多个目标与里程碑
├─ 成果证据 / PR / 隐藏进步
└─ 周期回顾与重校准

成长域
├─ 域切换器（只显示已激活域）
├─ 域概览与目标关联
├─ 专业计划与执行历史
├─ 域数据 / 洞察 / 安全
└─ 知识、派生内容与来源

我的
├─ 已确认画像与版本
├─ 数据与权限
├─ 调性与通知
├─ 离线 / 同步
└─ 导出 / 删除 / 账号
```

### 2.3 核心路径

**路径 1：首次建立身份与目标**  
价值预览 → 身份渐进追问 → 当前/目标身份草案 → 用户确认 → 候选成长域与目标 → 域内里程碑 → 进入旅程总览。

**路径 2：日常执行**  
今日 → 查看安排理由与状态 → 可选 Prime Pulse → 打开体能行动 → 查看动作/安全信息 → 执行 → RPE/疼痛/质量反馈 → 返回今日查看证据与调整。

**路径 3：查看长期迁移**  
旅程 → 当前身份与目标身份 → 选择成长域或目标 → 查看里程碑、成果证据与调整记录 → 周期回顾 → 确认或拒绝画像/目标重校准建议。

**路径 4：体能专业深扎**  
成长域 → 体能 → 域内目标 → 身体状态与专业计划 → 训练详情/动作学习/安全理由 → 执行历史与洞察。

**路径 5：知识进入实践**  
成长域 → 体能或相关域 → 知识/笔记 → 发起适用性评估 → 查看专家理由 → 用户逐条确认 → 渐进进入计划；或从训练卡的派生来源直接回溯父节点。

### 2.4 身份迁移如何可见

- 今日顶部不是泛化问候，而是“目标身份方向 + 今天这一步的作用”。
- 旅程首屏固定表达“当前身份 → 目标身份”，并把成长域、目标、里程碑和证据放在同一连续结构中。
- 每个目标显示其来源身份差距，避免目标成为孤立项目。
- 执行反馈用“形成了什么证据 / 哪个能力正在变化”替代纯完成百分比。
- 周期变化只生成候选叙事；用户确认后才成为新画像版本。

### 2.5 融合如何可见

- 今日在有真实证据时显示“今日融合”轻模块；样本不足时隐藏或使用学习态，不显示 0 分。
- 训练卡中的派生语言内容携带来源芯片，可回到训练、目标、笔记或书摘。
- 旅程以身份主题和目标关系展示跨域贡献，不以“活跃域数量”作为成就。
- 成长域内的知识注入入口始终与当前域/目标绑定，评估后由用户确认，不能静默进入计划。
- 同一时间槽仍只执行一个领域；融合表达知识和证据的流动，不暗示同时多任务。

### 2.6 优势

- 直接对应“今日循环 + 周期循环”，产品心智清晰。
- “旅程”天然容纳身份、目标、成果和重校准，避免把身份藏进“我的”。
- “成长域”可以为体能提供足够纵深，同时保留未来最多 3 域的同构空间。
- 一级导航从 v6 六项收敛至四项，降低移动端底部导航压力。
- 知识不再像独立内容产品，而是进入成长域和目标上下文。

### 2.7 风险

- “旅程”概念若文案和首屏结构不清，可能被理解为回顾或时间线，而非身份迁移中枢。
- “成长域”内部承载身体、训练、知识与洞察，二级信息架构要求较高。
- v6 用户需要适应“身体/训练/目标/知识”入口被重新归组。
- 如果今日页塞入过多身份、状态、计划和融合内容，会重新形成仪表盘堆叠。

### 2.8 对 v6 的迁移成本

**中等。**

- 今日：大部分保留，但需重排为“身份关联 → 安排理由 → 行动 → 证据”。
- 目标：并入旅程，补身份主轴和成果/重校准关系。
- 身体 + 训练：并入成长域中的体能空间，主要是路由和层级重组。
- 知识：并入对应成长域/目标上下文，同时保留来源回溯能力。
- 我的：保留治理类能力，移出不应由设置承载的身份迁移主内容。

## 3. 方案 B：身份主轴四 Tab（强主轴对照组）

### 3.1 一级导航

1. **今日**：今日安排、状态、执行与反馈。
2. **身份**：当前身份、目标身份、活跃域、身份差距、画像版本和迁移证据。
3. **计划**：跨域计划总览、域/目标筛选、专业方案、时间编排、冲突和执行历史。
4. **我的**：权限、数据、调性、通知、同步和账号治理。

目标、里程碑、成果和知识分别嵌入“身份”或“计划”的上下文，不设独立一级入口。

### 3.2 核心路径

**身份建立**  
身份 → 渐进追问 → 当前/目标身份 → 活跃域 → 每域候选目标 → 确认。

**查看迁移**  
身份 → 身份差距 → 选择域 → 目标/里程碑 → 证据与画像版本。

**执行计划**  
今日 → 今日主行动 → 计划详情 → 执行 → 反馈；或计划 → 域/目标筛选 → 专业方案 → 具体行动。

**处理多域冲突**  
计划 → 周期/周视图 → 冲突理由 → 一键采纳或我自己来；安全冲突转入不可覆盖的安全替代路径。

**知识注入**  
计划或域目标详情 → 关联知识 → 适用性评估 → 理由 → 用户确认。

### 3.3 身份迁移如何可见

- “身份”作为一级 Tab，拥有最高直接可发现性。
- 身份页按“现在是谁—想成为谁—差距—目标—证据—版本”纵向组织。
- 今日和计划中的每个行动均可回到身份差距或目标来源。
- 成果先进入身份页的证据区，用户确认后才改变画像版本。

### 3.4 融合如何可见

- 身份页展示多个活跃域如何服务同一目标身份。
- 计划页展示跨域共享的时间与能量预算，但每个时间槽仍为单域任务。
- 派生内容、来源芯片与知识注入仍长在计划/执行上下文。
- 今日只在有真实证据时呈现跨域回流，不常驻空壳模块。

### 3.5 优势

- 对“身份迁移是唯一最高主轴”的表达最直接。
- 用户能明确区分“我为什么改变”和“系统如何安排”。
- 跨域计划、冲突和统一时间编排在同一入口中更集中。
- 一级导航同样保持四项，结构克制。

### 3.6 风险

- “身份”容易被误解为静态个人档案，需要大量内容设计证明它是动态迁移系统。
- 体能专业深度被放入“计划”后，身体状态、动作库和知识的可发现性可能下降。
- “计划”可能让产品看起来像高级日程/任务工具，削弱成长 OS 的差异。
- 目标既服务身份又驱动计划，跨两个一级区域，信息归属和编辑入口容易重复。

### 3.7 对 v6 的迁移成本

**中高。**

- 今日可部分保留。
- 身份需要从“我的”与目标能力中重构为新的一级中枢。
- 身体、训练、目标、知识均需重新分配到身份或计划，页面归属变化较大。
- v6 既有体能心智连续性较弱，需要更强的导航引导和回归测试。

## 4. 方案 C：v6 收敛五 Tab（低迁移成本对照组）

### 4.1 一级导航

1. **今日**：今日状态、安排理由、行动与反馈。
2. **身体**：身体状态、准备度、恢复、负荷、安全和体能数据。
3. **训练**：体能专业计划、动作学习、执行与回顾。
4. **成长**：身份迁移、成长域、域内多个目标、里程碑、成果、知识与融合证据。
5. **我的**：个人治理与设置。

相较 v6，移除独立“目标”和“知识”Tab，将二者纳入“成长”；保留身体与训练的强入口。

### 4.2 核心路径

**身份与目标**  
成长 → 当前/目标身份 → 活跃域 → 域内目标 → 里程碑。

**今日训练**  
今日 → 安排理由/状态 → 训练 → 动作详情 → 执行 → RPE/疼痛 → 回顾。

**身体风险与调整**  
身体 → 状态/恢复/负荷 → 查看风险证据 → 安全替代或降载 → 回到今日/训练。

**知识与融合**  
成长 → 选择域/目标 → 知识与笔记 → 注入评估；或训练卡 → 派生来源 → 成长中的父节点。

**身份重校准**  
成长 → 成果与周期回顾 → 画像候选变化 → 用户确认 → 新版本。

### 4.3 身份迁移如何可见

- 今日顶部持续显示目标身份关联。
- “成长”首屏以当前身份、目标身份和成长域为第一层，目标和知识位于其下。
- 训练完成后回流“成长证据”，并在成长页形成周期叙事。
- “我的”不承载核心身份迁移，只保留治理信息。

### 4.4 融合如何可见

- 成长页统一呈现域关系、来源芯片、知识注入和融合证据。
- 今日和训练卡继续展示派生内容及来源。
- 身体和训练共享体能目标上下文，但不会新增独立融合 Tab。

### 4.5 优势

- 最大程度保留 v6 已验证的体能专业入口和用户路径。
- 身体状态与训练执行分离，适合早期体能单域深扎。
- 目标与知识从两个独立 Tab 收敛为“成长”，底部导航从六项降至五项。
- 原型到 Flutter 的页面与路由改造相对较少。

### 4.6 风险

- 一级导航仍有两个体能专属入口，产品容易继续被归类为健身 App。
- “成长”同时承载身份、目标、知识、成果和融合，可能成为大杂烩。
- 当第二、第三成长域逐渐变深时，“身体/训练”作为固定 Tab 会破坏多域同构。
- 五 Tab 在小屏和大字号下仍需重点验证；身份主轴显著度弱于方案 A/B。

### 4.7 对 v6 的迁移成本

**低至中等。**

- 今日、身体、训练、我的可保留主要导航心智。
- 目标与知识合并到成长，需要内容迁移和入口重命名。
- 身份迁移能力仍需新增，但可集中在成长页补齐。
- 短期成本最低，长期若扩展多域，可能产生二次 IA 重构成本。

## 5. 推荐与待验证决策

### 5.1 推荐方案

推荐以**方案 A：今日 / 旅程 / 成长域 / 我的**作为下一轮低保真原型和 Spec 验证主方案；同时保留方案 B 作为“身份显著度上限”对照，保留方案 C 作为“v6 迁移成本下限”对照。

推荐不是冻结结论，理由是当前仍缺少：

- 单域、双域、三域且每域多目标的真实内容压力测试；
- 今日、旅程与成长域之间的跨层返回规则；
- 体能内身体数据、训练、知识和洞察的二级 IA；
- 身份进度、成果证据和融合学习态的最终表达；
- 页面、状态、权限、离线与安全场景对应的 Spec；
- 可用性验证数据。

### 5.2 Spec 阶段应锁定的内容

1. 一级 Tab 名称、数量、顺序和未登录/未完成身份定义时的状态；
2. 路由与返回栈；Tab 内二级导航及深链规则；
3. 身份、成长域、目标、里程碑、计划、行动、知识和成果的唯一归属与交叉入口；
4. “最多 3 个活跃域、每域多个目标”的创建、暂停、归档和超限路径；
5. 今日页首屏内容优先级及空、错、加载、离线、权限拒绝和安全否决状态；
6. 体能专业空间的身体/训练/知识/洞察二级结构；
7. 身份迁移、目标邻近度、融合证据的数据充分度和禁止显示条件；
8. Prime Pulse 非阻断路径及长按/滑动的无障碍等效操作；
9. 从 v6 到新 IA 的路由映射、埋点迁移与回归范围。

### 5.3 推荐验证任务

以相同内容分别制作方案 A、B、C 的低保真可点击原型，至少覆盖：

- 新用户从愿望表达走到身份、域、目标与里程碑确认；
- 单域用户完成一次体能训练并提交 RPE/疼痛反馈；
- 双域或三域用户查看同一身份下的多个目标；
- 用户从训练卡回溯派生内容来源；
- 用户将知识发起评估并选择采纳、改造采纳、暂缓或不采纳；
- 用户查看周期成果并确认或拒绝画像变化；
- 安全事件发生时查看理由、替代与复评条件；
- 权限拒绝、数据不足、离线和服务超时的降级路径。

建议用任务成功率、错误入口次数、返回迷失率、首次理解准确率和关键路径步数作为比较依据，不用“看起来喜欢哪套”替代结构判断。

## 7. 评审材料排版简化建议

### 7.1 第一页只放决策摘要

所有后续评审稿统一采用一页式开头，避免评审者先进入几十页模块细节。建议固定为：

| 区块 | 内容上限 |
|---|---|
| 本次要决定什么 | 3 项以内 |
| 推荐方向 | 1 句话 + 3 条理由 |
| 未决 P0 | 5 项以内 |
| 方案对比 | 1 张矩阵 |
| 不在本轮决定 | 3–5 项 |
| 下一门禁 | 谁在何时产出什么 |

第一页只允许出现“已确认 / 待确认 / 被阻塞”三类状态，不混入详细背景叙事。

### 7.2 P0 前置，不埋在文档末尾

将所有“确认后才能进入 Spec”的 P0 决策移动到执行摘要之后，按以下格式书写：

| ID | 决策问题 | 选项 | 推荐 | 不决定的影响 | 负责人 |
|---|---|---|---|---|---|
| P0-IA-01 | 一级导航采用哪一方向 | A / B / C | A，待验证 | 页面归属与路由无法冻结 | 产品负责人 |

详细论证放在后文并通过 ID 回链。这样可防止评审讨论大量细节后仍未处理真正门禁。

### 7.3 追踪矩阵提供简化视图

主评审材料不直接铺开“蓝图—原型—PRD—Spec—架构—测试”全字段明细。正文先使用四列简化视图：

| Requirement | 产品状态 | 设计/Spec 状态 | 证据或阻塞项 |
|---|---|---|---|
| 身份迁移主轴 | 已确认/待确认 | 候选/已冻结 | 文档或决策 ID |

完整矩阵作为附录或独立真源，保留版本、负责人、验收场景、架构映射和测试 ID。正文只展示本次受影响项、阻塞项和状态变化项。

建议状态词统一为：

- `Confirmed`：产品原则已确认；
- `Candidate`：有候选，尚未冻结；
- `Blocked`：存在前置决策或证据缺口；
- `Spec Ready`：产品规则已可操作化；
- `Implemented`：已实现但未必验收；
- `Accepted`：已通过验收。

不得用“基本完成、差不多确认、原则可行”等模糊状态。

### 7.4 术语统一

建议建立评审稿页首术语卡，只保留一套名称：

| 统一术语 | 使用边界 | 避免混用 |
|---|---|---|
| 身份迁移 | 当前身份走向目标身份的最高主轴 | 身份升级、人格升级、画像成长混用 |
| 成长域 | 用户主动激活的成长领域，最多 3 个 | 维度、赛道、模块、频道混用 |
| 目标 | 某成长域内、由身份差距推导的目标；每域可多个 | 将“3 域”误写为“3 目标” |
| 身份画像 | 用户确认后的身份描述与版本 | 将系统推断直接称为用户画像事实 |
| 身份证据 | 执行、成果或趋势对身份变化的可追溯支撑 | 完成数、打卡数 |
| 融合 | 内容、知识、素材和证据在域间有来源地流动 | 同时多任务、跨域打卡 |
| 安全否决 | 阻止危险具体方案并给替代 | 阻止用户保留目标 |
| Prime Pulse | 可绕过的启动仪式 | 签到、准备度、训练解锁混用 |

### 7.5 版本统一

建议所有评审材料统一使用：

```text
文档名：PrimeAtlas-{主题}-{YYYY-MM-DD}.md
版本：v{主}.{次} Review / Candidate / Frozen
状态：Draft / In Review / Blocked / Approved
产品范围版本：Full PRD 或具体 Release Spec
体验参考：prototype-v6（仅参考，不代表冻结）
决策基线：Decision Log ID + 日期
```

规则：

1. 不再复用旧 S1 名称承载不同范围；
2. “v6”只指体验原型版本，不代替产品版本或 Spec 版本；
3. `Review`、`Candidate` 与 `Frozen` 必须区分；
4. 文档有修改时更新版本和状态，不只修改正文；
5. 被后续决策替代的文件保留历史，但页首标明 `Superseded by`；
6. 所有候选 IA 在 Spec 决策前持续标记为 `Candidate`。

## 9. 最终结论

PrimeAtlas 的一级导航不应直接复刻 v6 的工具分类，也不应为了强调身份而牺牲体能专业深度。当前更平衡的候选是：

> **今日负责把身份目标带入现实行动；旅程负责展示长期身份迁移；成长域负责专业深扎与融合流动；我的负责数据和个人治理。**

因此，建议以方案 A 进入下一轮低保真验证与 Spec 编写，但在完成多域多目标压力测试、关键路径验证、状态/权限/安全/离线映射前，**不得将其写成最终 IA 或直接冻结路由**。
