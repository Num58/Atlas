## 0. 文档目的与裁决规则

本 PRD 将完整蓝图、v6 原型、产品重校准结论和专项决策合并为可评审的产品契约草案，回答四个问题：

1. PrimeAtlas 最终必须提供哪些真实可用能力；
2. 各能力为什么存在、由谁使用、遵循什么业务规则；
3. v6 原型已经验证了什么，仍缺什么；
4. 哪些口径必须由产品负责人拍板，不能由研发自行猜测。

裁决顺序：产品定位与铁律 > 用户主权与安全 > 已确认交互真源 > 可验收性 > 实现便利。蓝图与原型冲突时，记录冲突并显式决策，不静默选择。全量表示终局功能全部纳入，不代表首版一次性交付；每个小版本仍需是可体验、可验证的完整子集。

## 1. 产品定义

### 1.1 一句话定位

**PrimeAtlas 是以“成为谁”为北极星的个人成长操作系统。用户定义当前身份和目标身份，系统将身份差距转化为目标、里程碑和专业成长计划，并根据执行反馈、能量、场景和成果持续适配。**

### 1.2 核心价值主张

> 不是管理你的任务，而是管理你成为谁。

系统不以“今天是否打卡”为中心，而回答：用户想成为谁、还差什么、今天为什么这样安排、现实变化后如何继续、哪些成果已经成为新的身份证据。

### 1.3 核心问题

处于身份转型期、同时经营多个成长方向的人，难以独立将模糊愿望、专业方法、有限时间、波动能量、身体安全和现实场景组织成一条连续、可信、可执行、不会因中断而失败的路径。现有单域 App、打卡工具、通用计划工具和通用 AI 都无法同时解决身份、专业性、跨域编排、安全和持续适配。

### 1.4 产品设计铁律

| ID | 铁律 | 必须做到 | 禁止做法 |
|---|---|---|---|
| P-01 | 目标驱动 > 打卡驱动 | 核心反馈解释目标邻近度 | 连续天数主视觉、断签惩罚 |
| P-02 | 成果驱动 > 压力驱动 | 主动发现 PR 与隐藏进步 | 排行榜、红点焦虑、失败羞辱 |
| P-03 | 身份认同 > 意志力消耗 | 强化正在形成的身份证据 | 只强调坚持、懒惰和补偿 |
| P-04 | 自适应 > 固定计划 | 场景和状态变化后仍可继续 | 中断即失败、全部手工重排 |
| P-05 | 多域共享统一框架 | 统一身份—目标—计划—成果模型 | 各领域互不相通 |
| P-06 | 专业可靠 | 专业设计、交叉评审、安全审查、理由可查 | 随机组合、无热身拉伸、黑盒结论 |
| P-07 | 用户主权 | 普通建议可采纳、调整或覆盖 | 强制普通目标取舍 |
| P-08 | 安全优先 | 危险方案必须否决并提供替代 | 迎合用户继续输出高风险方案 |
| P-09 | 数据克制 | 不足时明确未知并保守降级 | 伪精确百分比和无样本预测 |
| P-10 | 默认私密 | 分享由用户主动触发 | 默认公开和社交比较 |

### 1.5 非目标

- 不做公域社区、动态流、排行榜和好友竞争；
- 不做连续打卡、打卡群和断签惩罚；
- 不做装备商城、教练交易和知识付费商城；
- 不做通用 To-do、日历或企业 OKR；
- 不做医疗诊断、疾病治疗决策或心理评估；
- 不在训练组间、跑步中或阅读中强塞另一领域任务；
- 初期不以开放聊天机器人为主界面；
- 不向普通用户暴露未经整理的模型原始推理链。

## 2. 目标用户与 JTBD

### 2.1 核心用户

- 有明确或模糊的身份跃迁愿望；
- 同时经营 1–3 个活跃成长域；
- 时间、精力、地点和身体状态波动明显；
- 重视专业性、解释性和数据主权；
- 反感打卡焦虑、羞辱式督促和黑盒计划。

典型用户包括：希望成为国际篮球教练的程序员；希望在工作与生活节奏变化中恢复运动、工作能力和精力节奏的健康成年人；希望将专业阅读转化为实践能力的进阶学习者；只聚焦一个领域的单域用户。

### 2.2 首发适用人群与训练方案筛查

- 首发仅面向 **18 岁以上普通健康成年人**；
- 未成年人、孕期、急性疼痛、明确疾病、术后康复等身份/场景，不直接生成普通精细训练方案；
- 在生成任何训练方案前，用户必须通过**主动、非默认勾选**确认自己不属于上述身份/场景，默认值必须为未勾选；
- 用户命中任一身份/场景、拒绝确认或无法确认时，系统不得输出普通精细训练方案，应转为保守提示、停止相关危险建议，并建议寻求适当的专业评估或服务；
- 该确认仅是首发适用范围筛查，不是免责条款、医疗判断或安全通过证明，不替代计划生成、自动调整、训练启动和执行过程中的持续安全审查。

### 2.3 核心 JTBD

当我想成为一个新的自己，但不知道如何将多个目标、专业方法、时间和现实限制组织起来时，我希望系统理解我的身份差距，生成可信计划，帮助我每天执行，并在现实变化时持续调整，让我看到正在形成的能力和身份证据。

### 2.4 辅助 JTBD

- 目标模糊时，通过渐进追问说清真正想要什么；
- 多目标冲突时，理解权衡并选择推荐或自己编排；
- 伤病或负荷异常时，阻止危险方案但不否定目标；
- 读到有用知识时，判断能否安全进入实践计划；
- 状态差、出差或加班时，获得可继续的替代方案；
- 进步缓慢时，发现自己没意识到的变化；
- 改变方向时，保留历史并重新校准。

## 3. 产品循环与端到端旅程

### 3.1 主循环

身份定义 → 目标推导与里程碑 → 交叉融合与多 Agent 决策 → 专业计划 → 每日执行与反馈 → 成果和洞察 → 身份与目标重校准。

### 3.2 日循环

今日摘要 → 可选 Prime Pulse 启动 → 能量/恢复输入 → 查看计划与调整理由 → 执行 → RPE/疼痛/质量反馈 → 参数回流 → 身份证据反馈。

### 3.3 周期循环

周期数据汇总 → Agent 复盘 → PR/隐藏进步检测 → 计划迭代 → 下周期预览与承诺 → 用户确认画像变化。

### 3.4 重大变化循环

变化声明或检测 → 影响评估 → 暂停/归档/降级/替换/重排建议 → 用户确认 → 冲突和安全重审 → 新计划与必要的画像版本。

### 3.5 知识流动循环

导入知识 → 提取建议 → 加载画像/计划 → 专业和安全评估 → 采纳/改造采纳/暂缓/不采纳及理由 → 用户逐条确认 → 渐进融入 → 执行结果回流。

## 5. 全量功能需求 A–K

## 5A. A 身份系统

### A1 身份定义

- 用户以词、短语或自然语言自由表达，不要求先选标签；
- 五阶段流程：种子收集、维度展开、关联发现、深度追问、画像合成；
- 所有追问可跳过，用户说“够了”立即停止；建议 3–6 轮但不能成为门槛；
- 输出当前身份、目标身份、成长域、约束、跨域关系的可编辑草案；
- 用户确认后才持久化为有效画像；系统推测不得写成用户事实；
- 数据必须区分用户自述、客观数据、系统推断和模型生成。

**验收**：用户能从自由输入走到身份草案并自由修改；取消、跳过或拒绝无惩罚；未确认内容不进入正式画像。

### A2 身份和内部标签管理

- 用户管理自我描述、当前角色、目标角色和活跃域，而不是内部标签表；
- 内部使用领域、身份、特质三层标签进行推荐和规则判断，默认不可见；
- 修改身份时展示可能受影响的目标、计划和融合关系；
- 变更确认后创建新版本，并保存原因、来源和时间；
- 未激活域不展示、不保存占位数据、不进入 Agent 上下文或指标分母。

### A3 身份进度

- 身份进度由已激活域和已确认目标的证据计算；未激活维度权重为 0，分母动态归一；
- 展示参与维度、公式版本、数据窗口、更新时间、数据充分度和证据；
- 可量化目标可显示进度；不可量化目标优先展示阶段和证据，不强造百分比；
- 源数据更正后必须重算，并保留旧结果审计记录；
- 冷启动和样本不足显示“学习中/数据不足”，不显示 0 分或预计完成日期。

### A4 过渡态叙事与画像版本

- 仅在有可说明变化时生成候选叙事；
- PR、目标完成和长期趋势只能产生画像更新建议，不能自动改写画像；
- 用户确认后生成新的 `portrait_version`，历史可查看和恢复；
- 身份阶段可采用已确认的启程者、践行者、进阶者、掌控者，但需基于明确证据而非时长自动升级。

## 5B. B 目标系统

### B1 目标推导

- 根据当前身份、目标身份、能力差距、约束和用户优先级生成候选目标；
- 每个候选目标展示来源身份、差距、基线、目标状态、证据和建议原因；
- 用户可修改、删除、补充、改变优先级和期限；
- 目标确认后才进入计划生成；
- 无专业基准或数据不足时，输出需验证的探索目标，不给伪精确终点。

### B2 里程碑拆解

- 每个里程碑包含起点、目标值/状态、周期、证据来源、完成规则和依赖；
- 支持量化里程碑和行为/作品/能力证据型里程碑；
- 目标修改后，关联里程碑和计划保持一致更新；
- 里程碑完成可触发 PR 候选，但不能自动成为身份升级。

### B3 目标仪表盘

- 展示各目标状态、进度/阶段、健康度、风险、里程碑和关联计划；
- 可钻取查看公式、证据、调整记录和 Agent 理由；
- 允许查看活跃、暂停、调整中、完成和归档目标；
- 目标健康度必须有可测试定义，不能只显示神秘分数。

### B4 生命周期与重校准

状态至少包括：草稿、待确认、活跃、暂停、调整中、完成、归档；删除与归档语义必须不同，历史执行记录不得误删。

- 用户主动改变方向、重大生活变化、长期停滞、风险事件均可启动重校准；
- 系统给影响分析和建议，用户确认后生效；
- 暂停后释放时间预算但保留数据，恢复时重新评审；
- 已确认业务口径：最多同时主动编排 3 个成长域，每域可包含多个目标；超限提供聚焦建议，不静默丢弃。

## 5C. C 计划编排系统

### C1 计划生成

输入包括：已确认目标、里程碑、时间模板、能量曲线、场地设备、身体限制、当前计划和数据充分度。

输出包括：周期计划、周计划、日计划、任务、目标关联、依据版本、参与 Agent/规则和变更理由。

- 数据不足时必须采用保守模板或请求补充信息；
- 生成训练方案前必须完成 2.2 的主动、非默认适用范围确认；未成年人、孕期、急性疼痛、明确疾病、术后康复，或用户拒绝/无法确认时，不得输出普通精细训练方案，只能提供保守提示和专业转介；
- 上述确认不替代持续安全审查，未通过安全审查的方案不得输出为可执行计划；
- Agent 超时、无网络或数据矛盾时，不得伪装成完整专业方案；
- 用户可预览和有限微调，重大变化需重审。

### C2 场景自适应

支持正常日、加班、出差、生病/恢复、假期、碎片日、低能量和高能量等场景。

- 场景来源必须标明：用户手动、日历、位置、健康数据或系统推断；
- **可静默执行**：低风险、可逆，且不改变目标、里程碑、身份画像、安全等级、跨域优先级或计划主结构的小幅调整，例如已确认时间窗内重排、等价任务替换、既定专业边界内的容量微调；生效后必须显性通知回顾并支持回退；
- **必须事前确认或进入安全通道**：目标/里程碑/画像变化、跨域优先级变化、计划主结构改变、超出既定专业幅度、引入新的数据用途，或任何安全相关结论；画像变化一律只生成候选并由用户确认；
- 体能容量调整沿用既定边界：单次上调不超过 15%、单次下调不超过 20%；具体子场景阈值与规则版本仍需在 Spec 可测试化，但不得扩大产品层边界；
- 所有调整展示触发依据、前后差异、影响、版本和撤销/回退能力；
- 自动检测错误时用户可纠正。

### C3 时间槽编排

- 同一时间段只安排一个领域任务；
- 高能量匹配高强度任务，低能量匹配恢复或低负荷任务；
- 领域切换留有缓冲；
- 支持周视图、时间模板、拖拽调整和日历授权；
- 目标优先级、进度、能量和新鲜度可作为权重，但公式需版本化；
- 日历或位置权限拒绝后仍支持手动时间模板。

### C4 冲突检测与调解

普通冲突包括方向、时间、能量和领域优先级冲突：

- 展示冲突类型、受影响目标、代价和建议；
- 必须提供“一键采纳”和“我自己来”；
- 用户可保留全部目标，普通冲突不得硬阻断；
- 采纳、忽略和手调均记录原因但不产生惩罚。

安全冲突包括急性疼痛、旧伤禁忌、孕期禁忌、负荷超线等：

- 可阻止系统生成、启动或继续危险具体方案；
- 不提供“仍按危险方案执行”；
- 必须给证据、规则版本、命中原因、替代/降载/暂停方案和复评条件；
- 目标仍可查看、保留和管理。

### C6 专业计划标准

体能主训练强制结构：一般热身 → 动态拉伸 → 神经激活 → 专项准备 → 爆发/速度 → 复合力量 → 辅助/孤立 → 核心/纠正 → 静态拉伸/筋膜放松/呼吸。

专业规则：

- 动作顺序遵循高神经需求到低神经需求、多关节到单关节、大肌群到小肌群；
- 热身和拉伸与主训练内容匹配；
- 全天覆盖晨间激活、午间恢复、晚间主训练、睡前放松，但没有合适时间时不能为了“覆盖”强塞任务；
- 单次上调不超过 15%，单次下调不超过 20%，ACWR 上限按冻结规则执行；
- 连续过量触发减载；数据不足时不得用 ACWR 伪判安全；
- 所有安全和专业规则须有来源、版本、适用人群和例外。

**计划输出门禁**：结构、动作逻辑、容量、热身、拉伸、安全审查及相关领域评审任一未通过，计划不得以“已审核”状态输出。

## 5D. D 执行系统

### D1 今日视图

- 第一眼展示今天为什么这样安排、与哪些目标相关、有什么调整和风险；
- 今日摘要应先读本地数据，不等待 Agent；
- 信息分层：状态与主行动优先，准备度、ACWR、身体快照和详细理由按需展开；
- 不以“今日完成百分比”制造失败压力，可改为已完成证据和剩余建议；
- Pulse 未执行时仍可查看计划和安全信息。

### D2 执行追踪

- 支持未开始、进行中、暂停、跳过、完成、失败、撤销等明确状态；
- 支持开始、暂停、完成、误操作恢复和跨日处理；
- 支持自动数据与手动记录，但必须标明来源；
- 滑动完成必须有按钮替代；执行中杀进程后可恢复；
- 已缓存计划和动作说明离线可用。

### D3 反馈与容量自适应

- 完成动作后采集 RPE、疼痛、完成质量、负荷和原因；
- 跳过 RPE 时不得用“上次 RPE”伪装为本次实测，可记录为缺失；
- 训练前恢复、训练中反馈、训练后综合判断形成调整候选；
- 疼痛或异常 RPE 立即触发安全评估；
- 用户频繁跳过或搜索某动作可形成替换/教学建议，但不是自动认定能力不足。

### D4 训练达成追踪

蓝图提供计划完成率 35%、动作完整性 25%、负荷完成率 25%、节奏合规率 15% 的体能样例。全量产品需：

- 明确该公式只适用于有相应数据的体能任务；
- 无传感器时动作完整性和节奏主要基于用户操作/自评，不能称自动识别；
- 跳过热身、拉伸、组数不足和负荷偏低触发询问与风险提示，不触发羞辱或补偿性训练；
- 不同领域使用统一心理框架，但允许专业指标不同。

### D5 动作学习与视频

三层学习：训练中 3 秒检查点、训练前约 30 秒预习、动作库 2–3 分钟完整教学。

- 视频需说明来源、版权、审核状态、更新时间和适用限制；
- 支持关键帧、正/侧视角、节奏、常见错误、退阶和进阶方案；
- 视频不可用时有文本/图片替代，不影响当前训练计时；
- 支持按动作、肌群和器材搜索；
- 训练中检查点只服务动作质量，不在组间强塞其他领域内容。

### D6 Agent 反馈闭环

执行记录形成结构化报告，供领域、安全、康复、睡眠和时间编排职责分析；每次修改计划保留数据依据、建议来源、用户确认和回退链路。

## 5E. E 成果系统

### E1 PR 检测

四类成果：突破型、节奏型、觉醒型、里程碑型。

- 每类必须有触发公式/规则、基线、证据、最小样本、去重窗口和撤销规则；
- “完成全部任务”不能随机等同 PR；
- 节奏型 PR 不以连续天数定义，应体现长期稳定、现实中断后的恢复和有效节奏；
- 觉醒型 PR 必须由用户确认，不由模型单方面定义认知变化；
- 跨域 PR 不做排行榜式横向比较。

### E2 PR 庆祝

- 庆祝强度按成果类型和用户偏好适配；
- 反馈关联目标和身份证据；
- 不出现排名、streak、同期用户比较或羞辱；
- 支持减少动态效果和关闭音效；
- PR 数据更正或撤销后，关联进度、画像建议和报告同步修正。

### E3 成长回顾

- 支持周、月、季、年及目标周期回顾；
- 用故事线组织目标变化、成果、恢复、场景适配和下一步，而不是数据堆砌；
- 展示数据充分度和推断边界；
- 用户可纠正误判并决定是否产生画像候选更新。

### E4 成果分享

- 默认私密，用户主动选择分享；
- 分享前预览字段，可排除健康、伤病和具体负荷；
- 不包含连续天数、排名和他人比较；
- 支持生成卡片/图片/链接及撤销；
- 分享能力是否首发延期需由产品负责人重新确认，但终局范围保留。

## 5F. F 能量系统

### F1 能量状态感知

- 采集主观能量，并可选接入睡眠、HRV、静息心率、酸痛和恢复数据；
- 每项标记来源、时间、有效期和可信度；
- 健康数据授权与其他授权分开；拒绝授权后仍可用主观输入；
- 主客观冲突时展示差异和保守建议，不把设备分数当真相；
- 跨设备指标不可直接比较时必须提示口径差异。

### F2 准备度与计划影响

- 准备度与长期目标势能、训练启动状态分开；
- 数据充分时可形成评分，公式、特征、权重和窗口可查；
- 数据不足时显示高/中/低/恢复中等等级或“数据不足”，不显示伪精确 87%；
- 不同区间对应的强度调整范围和人工覆盖规则可测试；
- 安全相关异常不能被用户普通覆盖绕过。

### F3 恢复建议

- 输出主动恢复、完全休息、轻度活动等建议；
- 休息是计划的一部分，不产生惩罚；
- 建议不越界为医疗诊断；危险或持续异常时提示停止相关方案并寻求专业评估。

## 5G. G Prime Pulse 与仪式

### G1 每日启动仪式

- v6 交互真源为长按，产品建议阈值确认为 **500ms**；后续 Spec 只负责定义容差、状态与测试口径，不得回退为训练硬门槛；
- 定义按下、提前松手、重复触发、切后台和失败状态；
- 必须提供单击按钮或辅助技术等效入口，功能与长按完全等价；
- Pulse 可绕过，是可选启动仪式，不得阻止查看或执行安全计划；
- 长期目标势能、今日准备度和启动仪式是三个独立概念，不得用一次长按或同一分数混合表示；
- 长按只代表用户选择开始，不代表准备度或健康评估完成。

### G2 火焰/势能

- 火焰代表长期目标接近度和近期趋势，不代表连续天数；
- 目标接近度必须来自可解释指标，数据不足时采用学习态；
- 休息日切换养护状态，不下降、不催促；
- 不把火焰弱化解释为用户失败。

### G3 目标承诺仪式

- 周期回顾、下周期预览和用户主动承诺组成一次仪式；
- 用户可调整、延期或暂不承诺，无惩罚；
- 承诺不替代对重大计划变化的具体确认。

### G4 休息日仪式

- 无任务催促和补偿性学习；
- 可查看数据和回顾；
- 明确恢复对目标的价值；
- 结束后按计划恢复，不用“补回”休息日。

## 5H. H 调性系统

### H1 调性选择

四种调性：热血、专业、陪伴、严厉；默认专业。

- 用户可查看示例和切换；
- 同一 `tone_version` 覆盖文案、视觉、动效、触感、推送和 Agent 解释；
- 调性只改变表达，不改变事实、权限、安全结论和功能可用性；
- 已生成历史内容不回写；切换确认后，后续输出立即按新 `tone_version` 全局一致生效，进行中的既有内容不回写。

### H2 全层适配

- 使用语义 Token 单一真源，不做四套独立 UI；
- 任一调性都满足可读性、对比度、动态字体和减少动效；
- 推送受通知授权、免打扰和频率上限约束；
- 同一会话不得局部漂移。

### H3 严厉型红线

严厉型仍禁止羞辱、威胁、罪感和失败标签。蓝图中的“别找借口”“别偷懒”“终于突破”等示例需重写，不能进入正式文案库。系统不得用调性扩大催促频率到制造焦虑。

### H4 切换健康带宽

已有 S0 支持健康带宽和冷却提示。产品层冻结：冷却仅用于温和提示用户维持一致体验，不得以 `blocked_by_system` 阻止用户切换；同一会话中，切换确认后的后续输出必须统一采用新 `tone_version`，进行中的既有内容不回写。平台状态机、提示频率与异常恢复留待 Spec 可测试化。

## 5I. I 数据与洞察

### I1 个人数据中心

- 聚合身份、目标、执行、能量、成果和融合证据；
- 数据按域、目标、周期和来源筛选；
- 展示更新时间、来源、有效期、可信度和更正入口；
- 未激活维度完全不出现。

### I2 隐藏进步

- 定义观察窗口、最少样本、效应阈值、置信阈值和冷却期；
- 每条洞察带证据和误判反馈入口；
- 不能把随机波动包装成惊喜；
- 数据不足显示“尚不足以判断”。

### I3 趋势预测

- 支持目标预计区间、风险和趋势变化，不承诺精确日期；
- 展示模型/公式版本、误差范围和样本充分度；
- 冷启动、数据中断和目标变化后重新建立基线；
- 用户修正源数据后重算。

### I4 健康报告

- 监测训练负荷、恢复、学习负荷和领域资源分配；
- “领域均衡度”只描述用户主动激活域，不惩罚单域用户；
- 红黄绿状态须有阈值和证据；
- 健康报告是风险提示和成长质量报告，不是医疗报告。

## 5J. J 基础能力

### J1 账号与身份

- 支持游客、本地账号态、注册、登录、登出、注销和账号切换；
- 游客本地数据到注册账户的合并规则明确；
- 登出或切换账户后不得泄露上个账户数据；
- 支持 Apple、Google、手机号的具体范围后续按发布地区冻结。

### J2 内容管理

- 管理动作资料、书摘/笔记、领域术语、模板和来源；
- 所有专业内容标记版权、审核、版本和失效状态；
- 外链失效时提供替代，不把未经审核的网络内容当专业真源；
- 自研示例摘录作为早期知识注入种子已确认。

### J3 通知

- 通知授权、免打扰、类型开关和调性适配；
- 无红点强迫和未完成羞辱；
- 休息日不推任务催促；
- 安全通知与普通提醒分级。

### J4 隐私、安全与数据权利

- 身体/伤病/睡眠、日历、位置、知识内容、画像推断、通知、文件/相册、麦克风、摄像头分别说明用途并授权；
- 拒绝任一权限不阻断无关核心能力；
- 默认私密，支持查看、导出、更正、撤回和删除；
- 第三方模型处理必须独立 Consent，仅发送完成当前功能所需的最小必要数据并默认去标识化；健康、伤病、原始笔记/书摘等敏感内容不得因其他授权被捆绑发送；
- 用户数据默认不得用于训练通用模型；任何未来变化必须单独、主动 opt-in；
- 蓝图“所有数据 E2E 加密”需改为可验证的数据分级、传输/静态/端侧加密和服务端可见范围；
- 注销删除需定义本地、云端、索引、衍生画像和备份处理时限。

### J5 离线与同步

离线可读写：已缓存今日计划、开始/暂停/完成、RPE、疼痛、能量、动作说明。离线待处理：新目标生成、云端 Agent 评审、知识注入和未缓存内容。

- 本地操作有幂等事件 ID；
- 重连不得重复执行、重复 PR 或错账号归属；
- 多端计划冲突保留可解释版本，安全数据不得被旧版本覆盖；
- 同步目标 `<5s` 需补充起止点和 P95/P99；
- 技术名如 CRDT 不能替代用户可感知的冲突规则。

## 5K. K 交叉融合引擎

### K1 内容交叉

- 融合入口、来源芯片和已有融合证据默认可发现，不要求用户先开启独立“融合模式”；
- 领域内容服务语言/表达学习，例如篮球训练派生篮球英语；
- 只基于真实活跃域、目标和执行节点；
- 低置信内容不展示或允许用户校正；
- 每条派生内容带来源芯片，可回到训练、目标、笔记或书摘父节点；
- 数据采集和内容生成分别提供关闭/撤回能力；关闭后不得产生惩罚、红点、降权或完成压力，且不影响原领域核心体验。

### K2 知识注入

五阶段：知识提取 → 画像加载 → 多职责评估 → 综合决策 → 用户确认。

决策状态：采纳、改造采纳、暂缓、不采纳。

- 安全和康复判定为阻断级；
- 每条建议展示目标关联、身体匹配、安全、康复、时间、环境、恢复、计划兼容和优先级证据；
- 用户逐条确认后才进入计划；
- 采纳内容按适应、过渡、稳定三阶段渐进融入；
- 专家理由卡必须有结论、理由、依据、来源和下一步。

### K3 素材回流

- 训练和实践经历生成术语、表达、复盘或写作素材；
- 生成不等于必须学习，不产生新待办、红点或连续机制；
- 回流是否有效以真实下游复用为证据，而非生成卡片数量。

### K4 主题对齐

多个域的内容优先围绕共同身份主题选择，但保留基础能力训练。推荐必须有来源和用户相关性，不能仅靠标签猜测。

### K5 统一时间与场景编排

- 素材可以跨域，时间槽不能同时跨域；
- 加班、出差、生病、假期、碎片日和能量变化由 K/C 协同适配；
- 休息日不得偷换成学习日；
- 不强制每一天发生融合。

### K6 融合度量

过程指标采用：派生语言回流率、知识注入采纳率、读书到训练应用率、跨域融合留存率。

- 不数交叉次数、点击、生成卡、导入书和专家通过率；
- 早期代理指标不直接进入身份进度；
- n 不足时隐藏或显示学习态，不显示 0 分；
- 单域用户不显示融合模块、不占权重。

## 6. 多 Agent 专业决策体系

### 6.1 定义

12 Agent 是 12 类产品职责、权限和审查角色，不等于 12 个独立模型、12 个聊天机器人或 12 位真人专家。底层可由规则、专业内容、一个或多个模型及必要的人工审核组合承载，但不得因此丢失职责边界或越权。每个角色需定义：输入可用性、规则/模型性质、输出状态、写入权限、否决权限、失败接管和用户可见证据；面向用户的称谓不得暗示未实际提供的真人执业服务。

| ID | 职责 | 核心权限/边界 |
|---|---|---|
| FIT-COACH | 体能计划、动作、强度、周期 | 领域主导；开放生成需专业规则门禁 |
| SAFETY-OFFICER | 负荷、疼痛、动作安全 | 安全否决；规则优先、必须可追溯 |
| NUTRITIONIST | 一般营养信息和目标匹配 | 强建议；不诊断、不替代专业医疗营养服务 |
| SLEEP-EXPERT | 睡眠和恢复建议 | 强建议；不能直接比较不同设备评分 |
| REHAB-SPECIALIST | 伤病风险和安全替代 | 安全否决；不诊断或承诺康复效果 |
| LANG-COACH | 语言路径和素材设计 | 领域主导 |
| LANG-ASSESSOR | 难度和能力评估 | 强建议；非正式测试只能称估计等级 |
| READING-GUIDE | 阅读路径和书目编排 | 领域主导；受版权和访问权限约束 |
| KNOWLEDGE-TRANSFER | 知识到行动的适用性评审 | 强建议；必须有原文证据和用户确认 |
| TIME-ORCHESTRATOR | 多目标时间和场景编排 | 编排主导；不能假设总能自动检测场景 |
| TONE-ADAPTER | 全层表达一致性 | 强建议；不得改变事实和安全等级 |
| USER-PROFILE | 提供画像上下文和更新候选 | 建议权；不得静默改画像 |

### 6.2 协作流程

1. 新目标：画像上下文 → 领域职责并行设计 → 安全/康复串行审查 → 相关专业评审 → 时间编排 → 调性适配 → 用户结论页。
2. 日常调整：时间职责重算 → 受影响领域调整 → 安全检查变更部分 → 通知与回退。
3. 周度复盘：各职责分析 → 建议合并 → 冲突检测 → 普通仲裁或安全通道 → 周期回顾。
4. 安全事件：立即暂停相关危险动作 → 安全/康复评估 → 安全替代 → 领域方案调整 → 时间重排。

任何时间优化都不能跳过必要安全检查。蓝图“时间缺口≤30分钟时跳过评审”只能理解为跳过不相关的普通 Agent，不能跳过安全门禁。

### 6.3 仲裁

普通分歧可采用主导回应、多职责讨论、权重投票、用户决策四级递进。安全否决不进入普通仲裁，用户不能选择继续执行被判定为危险的原方案，但可选择不同安全替代、暂缓或重新评估。

### 6.4 用户可见性

- 默认：结论、参与职责、核心依据、数据充分度；
- 展开：结构化建议、分歧、采纳/拒绝理由、版本；
- 高级审计：规则命中、输入来源和变更链；
- 不向普通用户展示模型私有原始推理链；
- 模型名称不能代替专业解释。

### 6.5 失败与降级

- 计划生成、知识评估、安全检查分别定义超时和降级；
- 任何安全检查超时不得自动视为通过；
- 普通职责超时可使用已审核规则模板、延迟处理或返回部分结果；
- 不完整计划必须标为草稿/保守方案，不能标为专业审核完成；
- 模型切换不得改变安全和专业规则。

## 7. 数据、状态与权限契约（产品层）

### 7.1 数据来源标签

所有关键数据必须带：来源类型（用户/设备/第三方/系统推断/模型/专业内容）、采集时间、有效期、可信度、授权目的和可纠错性。

### 7.2 核心状态

| 对象 | 最低状态要求 |
|---|---|
| 身份画像 | 草稿、待确认、有效、被替代、归档 |
| 目标 | 草稿、待确认、活跃、暂停、调整中、完成、归档 |
| 里程碑 | 待开始、进行中、达成、未达成、被替代 |
| 计划 | 生成中、草稿、待审、安全否决、待确认、有效、过期、被替代 |
| 任务 | 未开始、进行中、暂停、跳过、完成、失败、撤销 |
| PR | 候选、待确认、有效、撤销、归档 |
| 注入建议 | 提取中、评估中、采纳、改造采纳、暂缓、不采纳、已融入 |
| 同步 | 本地待同步、同步中、成功、冲突、失败待处理 |

安全否决不能与用户拒绝普通建议共用同一状态。

### 7.3 版本与因果链

每次计划、画像、目标和重要指标变化记录：变化前后、触发数据、数据时间、规则/Agent、用户确认、版本、回退能力。源数据删除/更正后，相关进度、洞察、PR 和画像候选必须重算。

### 7.4 Consent

健康、伤病、睡眠、日历、位置、通知、文件/知识、麦克风、摄像头及第三方模型处理分别授权。第三方模型 Consent 不得被账号条款、健康权限或文件权限捆绑替代。撤回后停止新采集/发送，并说明既有数据和衍生结果如何处理。训练方案前的适用范围确认另行主动获取、默认未勾选；它不是数据处理 Consent，也不替代持续安全审查。

### 7.5 时间语义

后续 Spec 必须冻结统一时间精度、时区、夏令时、跨端排序和去重。现有 S0 事件/画像采用微秒时间戳，调性冷却使用毫秒；产品层要求是顺序稳定、单位明确、跨端可去重，而非混用未说明的 epoch 数值。

### 7.6 核心产品数据对象

以下对象必须在 Spec 中定义产品含义、所有者、来源、生命周期、敏感级别、版本和删除/归档语义：

1. 用户、云端账号、设备本地身份与会话；
2. Consent 与权限记录，包括目的、范围、接收方、有效期和撤回时间；
3. 身份草案、已确认画像、画像版本、活跃域和过渡态；
4. 目标、里程碑、基线、衡量口径、暂停与恢复条件；
5. 时间/场地/器材/日历/位置/健康/场景约束及其时效、来源、可信度；
6. 专业内容、动作、视频、书目、原文/笔记、术语、规则、版权和失效状态；
7. 计划、计划版本和行动单元，包含输入依据、确认状态、变更原因和回退能力；
8. 执行与反馈，包含原始发生时间、设备来源、RPE missing 状态和疼痛记录；
9. 主观能量、设备健康数据、平台数据和系统估算；
10. 融合父来源、派生内容、评审结论和采纳历史；
11. Agent 运行、参与职责、权限性质、输入、依据、充分度、降级和用户采纳；
12. PR/洞察候选、证据校验、去重、接受、忽略、更正和撤销；
13. 调性版本、通知权限、免打扰和反焦虑限制；
14. 分享副本、导出与删除请求；
15. 本地操作、同步状态、冲突和用户处理结果。

模型生成、系统推断和预测字段必须显式标识来源，不得伪装成用户事实或客观测量。滑动完成只证明用户声明完成，不等同于动作质量已被传感器验证。

### 7.7 Consent 详细规则

- 授权必须在对应功能首次需要时触发，不以前置总授权捆绑；
- 明示用途、数据范围、接收方、保留时间和拒绝后的最低可用体验；
- 用户拒绝某项权限，不得阻断与该权限无关的身份、目标、手动计划和执行记录；
- 撤回后立即停止新采集，并分别说明原始数据、衍生画像、索引、备份和第三方副本的处理；
- 健康、伤病、原始笔记/书摘发送第三方模型前需独立 Consent；第三方模型默认仅获最小必要、去标识化数据；
- 用户数据默认不得用于训练通用模型；新用途、扩大范围、更换接收方或拟用于通用模型训练必须重新主动同意；
- `system_auto` 只可产生画像候选，不得作为正式画像确认来源；
- 训练方案前必须主动、非默认确认用户为 18 岁以上普通健康成年人且不属于未成年人、孕期、急性疼痛、明确疾病、术后康复场景；命中、拒绝或无法确认时禁止普通精细训练方案并转保守提示/专业转介；
- 上述确认不替代持续安全审查，任何后续安全信号仍可否决或暂停方案。

### 7.8 离线与同步产品规则

- 真实执行记录采用追加语义，云端新计划不得覆盖用户已经发生的事实；
- 重复上传同一离线操作不得重复计数、产生 PR 或触发计划调整；
- 用户显式编辑优先于普通系统建议；Consent 撤回和安全限制以更严格、更新的状态优先；
- 多端画像冲突不得自动合并成正式画像，应保留候选供用户确认；
- 计划冲突同时保留本地实际执行依据和云端新版本，再决定后续生效版本；
- 跨时区和异常设备时间保留原发生时间、原时区和同步时间；
- 旧设备/缓存不得覆盖新安全限制、授权撤回或已确认画像；
- 同步失败不阻断本地执行，但必须展示未同步状态、重试入口和潜在影响。

## 8. 指标体系

### 8.1 用户价值指标

| 指标 | 产品定义 | 评审状态 |
|---|---|---|
| 身份进度指数 | 已激活维度的目标证据动态归一聚合 | 原则已定，公式待冻结 |
| 目标达成率 | 周期内有效达成的里程碑/可评估里程碑 | 需定义目标变更和取消口径 |
| 计划执行率 | 实际有效执行/当期有效计划 | 需排除被系统替代和安全暂停任务 |
| PR 频率 | 有效且去重的 PR/周期 | 四类型公式待冻结 |

### 8.2 维度指标

体能、语言、认知/阅读可使用不同专业指标，但都映射为基线、目标、证据、数据充分度和趋势，不要求指标名称完全相同。

### 8.3 融合代理指标

- 派生语言回流率；
- 知识注入采纳率；
- 读书到训练应用率；
- 跨域融合留存率。

这些指标早期为 Process/Guardrail，不进入北极星。

### 8.4 指标黑名单

不得把点击数、生成卡数、导入书数、活跃域数、专家通过率、连续天数和简单完成百分比包装成成长成功。

### 8.5 可解释性要求

身份、恢复、融合、准备度、预测等指标默认展示 **阶段 + 趋势 + 证据 + 数据充分度**。只有在公式/规则版本、参与数据、窗口、样本门槛、更新时间及误差/置信表达均满足对应显示条件时，才允许展示精确百分比；否则必须显示等级、区间、“学习中”或“数据不足”。所有结果提供纠错入口。PRD 评审后需为关键公式提供至少 3 组金标样例。

## 9. 非功能需求与质量门禁

### 9.1 性能

| 能力 | 蓝图目标 | PRD补充要求 |
|---|---|---|
| 冷启动 | <2s | 指定设备档位、P95；本地今日摘要先显示 |
| Pulse 反馈 | <100ms | 从触摸事件到可见/触觉反馈，P95 |
| 同步 | <5s | 定义起止点、正常网络、P95/P99 |
| 崩溃率 | <0.1% | 定义版本/用户会话窗口 |
| 计划生成 | 蓝图 60–90s | 与“<30s”建议冲突，需冻结进度、取消、超时和降级 |
| 日常调整 | ≤20s | 首屏不等待；安全审查不得被跳过 |
| 调性适配 | ≤5s | 模板或缓存降级 |

### 9.2 可靠性

- 弱网、超时和重复请求不得产生半计划、重复采纳或重复 PR；
- 核心执行不依赖实时网络；
- 数据冲突有状态和用户处理入口；
- 所有高风险规则有版本、测试样例和回滚能力。

### 9.3 可访问性

- WCAG 2.2 AA 和平台规范为基线；
- 触控目标 iOS ≥44×44pt，Android 建议 ≥48dp；
- 正文对比度 ≥4.5:1；
- 长按、滑动、颜色、发音、视频、震动均有等效替代；
- 支持屏幕阅读器、动态字体、键盘焦点、横屏/窄屏和减少动态效果；
- Pulse 火焰和 PR 动画避免闪烁风险。

### 9.4 隐私与合规

- 数据最小化、目的限定、默认私密、可撤回；
- 首发年龄与健康适用边界已确认：仅 18 岁以上普通健康成年人；未成年人、孕期、急性疼痛、明确疾病、术后康复不直接生成普通精细训练方案；
- 训练方案前必须主动、非默认确认；命中、拒绝或无法确认时转保守提示/专业转介，该确认不替代持续安全审查；
- 第三方模型最小必要、默认去标识化、独立 Consent，默认不用于通用模型训练；
- 受监管专业称谓、首发地区和数据删除 SLA 仍需在对应 Spec/合规流程前确定；
- 不承诺不可验证的“全量 E2E 加密 + 云端可计算”组合。

### 9.5 Agent 场景级 SLO

SLO 从用户发起操作至可见结果计时，Spec 必须补充 P95、超时状态、取消/后台等待和降级。安全正确性优先于速度。

| 场景 | 用户可见 SLO 建议 | 超时/不可用行为 |
|---|---:|---|
| TONE-ADAPTER 调性适配 | P95 ≤5秒 | 保持当前已确认调性，不混合调性，不改变专业结论 |
| 身份/目标引导单轮 | P95 ≤10秒 | 保存用户输入、允许稍后继续，未完成推断不写画像 |
| 日内重排/小幅领域调整 | P95 ≤20秒 | 保留最近有效安全计划，标记调整待处理，不清空今日页 |
| 安全/康复紧急触发 | 立即安全暂停；完整理由 P95 ≤30秒 | 保持暂停并提供保守替代，超时不得自动放行 |
| 完整计划与多职责评审 | P95 ≤90秒 | 展示进度并允许离开；只返回已过必要安全检查的部分/降级方案 |
| 知识提取与适用性评估 | 单个有界内容 P95 ≤60秒 | 保留待评估，不自动注入；来源/版权不明时停止使用 |
| 有界语言评测 | P95 ≤30秒 | 标为估计等级/未完成，不显示正式等级 |
| 睡眠/营养非紧急建议 | P95 ≤30秒 | 显示数据不足或一般建议，不生成精确健康结论 |
| PR/隐藏进步/趋势 | 非阻塞实时 | 在下次相关回顾可见；样本不足不生成或明确证据不足 |

外部模型、可穿戴、日历、位置和第三方内容必须分别定义四类降级：权限拒绝、数据不足/过期、服务超时/不可用、内容失效/授权撤回。

### 9.6 加密与服务端可见范围

- **仅设备可见**：若承诺端到端或仅设备可见，服务端/第三方模型不得读取明文，相应云端分析不可用或需用户主动解锁；
- **PrimeAtlas 服务端可见**：限于用户主动启用能力所必需的最小数据，说明用途、保留期和人工支持可见性；
- **第三方模型可见**：默认只发送完成本次请求所需的最小、去标识化内容；健康、伤病、原始笔记/书摘需独立授权；
- 用户数据默认不得用于训练通用模型；未来改变必须独立 opt-in；
- 日志和埋点不得记录完整健康信息、笔记原文、对话全文或身份叙事；
- 分享应生成独立最小化副本，接收者不能访问原始账号数据或自动获得后续更新；
- 导出应包含用户输入、已确认画像、目标/计划/执行和主要决策依据；删除需说明设备、服务端、索引、备份和第三方副本各自范围及时限。

## 10. v6 原型评审结论

### 10.1 已验证或较好表达

- 免登录价值预览和登录入口；
- `openAIGoalChat()` 五阶段目标画像方向；
- `startPulseHold()` / `handlePulseTap()` 长按启动；
- 今日页目标身体快照、准备度、ACWR 和全天训练时间线；
- 身体数字孪生、指标趋势、数据录入和 AI 引导测试；
- 训练场景、动作完成、RPE、动作视频、动作库和调整理由；
- 目标卡、里程碑、暂停/归档/分享及目标创建；
- 知识库、阅读、RAG、训练英语和知识注入入口；
- Profile、调性、主题、通知、同步、导出、隐私和模型设置入口。

### 10.2 仅为演示或存在产品冲突

- Pulse 当前显示“就绪·解锁训练”且锁住训练卡，与非阻断仪式方向冲突；
- Pulse 87、恢复 87%、今日 65%、数字孪生 87% 等为硬编码，公式不足；
- `startPulseHold()` 的 500ms 已被接受为产品建议阈值；原型仍缺提前松手、切后台、重复触发、失败态和按钮/辅助技术等效入口；
- `skipRPE()` 使用上次 RPE，违反数据真实性；
- 目标页“最多 3 个目标”与已确认“最多 3 个活跃域”冲突；
- 今日页和训练页进展百分比可能制造完成压力；
- 目标页主要是训练目标，身份主轴和多域关系不够可见；
- 六 Tab 同时放底部导航超出常规舒适数量，且“身体/训练/知识”容易形成工具拼盘；已确认六 Tab 仅为候选，不冻结为最终信息架构；
- 大量 emoji 作为功能图标、硬编码静态数据和视频模拟不可直接迁移；
- AI 模型切换直接开放给普通用户，可能与安全一致性冲突；
- Profile 中默认热血型，与已确认默认专业型冲突。

### 10.3 原型必须补齐的关键流程

- 完整身份确认、画像版本和过渡态叙事；
- 身份→多域目标→统一计划的全链路；
- 普通冲突双轨裁决与安全否决流程；
- 加班/出差/生病/重大变化及回退；
- Pulse 三语义拆分与非阻断入口；
- 融合来源芯片、专家理由卡、渐进融入和证据链；
- PR 回顾、隐藏进步、画像确认；
- 权限拒绝、无数据、弱网、离线、同步冲突和账号合并；
- 数据导出、删除、撤回授权和分享隐私预览；
- 屏幕阅读器、按钮替代、减少动效等无障碍路径。

## 11. 核心验收场景

| ID | Given | When | Then |
|---|---|---|---|
| AC-01 | 新用户自由描述目标身份 | 完成或提前结束追问 | 仅生成可编辑草案，确认后持久化 |

| AC-05 | 普通时间/方向冲突 | 系统检测 | 同时显示一键采纳和我自己来，目标不被删除 |

### 13.4 PRD Ready 硬门禁

以下 9 条是“相关需求可进入 Spec”的必要条件。2026-07-20 已确认的 12 项 P0 在产品层标记为 `Accepted`；其余可操作化门禁未满足时，受影响 Requirement 仍须保持 `Blocked for Spec`，不得通过解释性文字、原型演示或开发阶段补充来绕过：

1. **承诺可验收**：“永不断档”“无条件信赖”“所有细节专业”“自然融合”“自动理解一切”等表述，必须替换为可观察行为、覆盖范围、适用边界、失败条件和降级路径。
2. **数字有依据**：预计完成时间、恢复率、准备度、身份进度、融合度等数字必须有公式版本、数据源、样本门槛、窗口、误差/置信表达；条件不足时禁止伪精度。
3. **场景来源真实**：加班、出差、生病等不得笼统宣称自动检测，必须绑定用户主动声明或已分别授权的日历、位置、健康数据，并定义拒权、数据过期和误检测降级。
4. **完成质量有证据**：滑动完成与 RPE 只代表用户声明和主观反馈，不能自动证明动作/任务质量；必须冻结各域可用证据、来源标签和“不确定/数据不足”状态。
5. **成果规则完整**：隐藏进步、觉醒 PR 及其他成果必须有事件公式/可操作规则、最小样本、证据、去重窗口、误判反馈、撤销与重算规则。
6. **安全评审不可跳过**：蓝图“≤30分钟变更可跳过评审”只允许省略不相关的非安全复评；安全和康复检查永不因变更小、时间紧或 Agent 超时跳过。
7. **用户仲裁边界明确**：用户最终仲裁只适用于普通目标、时间和偏好冲突；安全否决不可由用户选择继续原危险方案覆盖。
8. **S0口径唯一**：`blocked_user=false` 仅适用于普通冲突；画像“自动更新”只能产生候选建议并由用户确认；调性“随时切换”必须与 cooldown/block 字段形成唯一产品规则。
9. **外部依赖四类降级完整**：任何依赖云端模型、可穿戴、日历、位置或第三方内容的需求，必须同时覆盖权限拒绝/撤回、数据不足/异常/过期、服务超时/限流/不可用、内容失效/下架/撤权/来源不可达四类场景。只具备正常路径不得标记 `PRD Ready`；安全能力不得因依赖失败而默认放行。

#### 外部依赖 Given/When/Then 横向检查表

| 类别 | Given | When | Then 必须定义 |
|---|---|---|---|
| 权限拒绝/撤回 | 用户未授权或已撤回健康、日历、位置、文件/笔记或第三方 AI 处理 | 相关功能尝试读取或生成结果 | 说明仍可用的手动/本地能力；禁止继续采集或发送；提示降级原因；处理旧缓存和衍生结果；提供重新授权但不强迫；记录 Consent 审计 |
| 数据不足/异常/过期 | 数据缺失、样本不足、设备异常、时效过期或来源冲突 | 系统计算准备度、风险、进度、预测或调整 | 禁止伪精度与“未发现风险=安全”；显示数据充分度/新鲜度；使用保守规则或用户输入；提供补录/纠错；记录使用了哪些数据和降级规则 |

| 内容失效/下架/撤权/来源不可达 | 动作视频、书摘、规则、术语或第三方资料不可用或授权失效 | 用户查看内容或系统继续引用其专业结论 | 失效内容不得继续作为有效专业依据；提供文本/图片/已审核替代；标记来源与失效时间；触发相关注入/计划重新评估；保留审计而不向用户展示失效内容 |
