# PrimeAtlas 产品理解与开发路线重新校准

> 日期：2026-07-20  
> 状态：产品真源校准完成，待据此重写下一小版本 Spec  
> 权威依据：`PrimeAtlas_完整需求蓝图_Final.md` + `primeatlas-prototype-v6.html` + 已确认 Phase1 产品战略文档

---

## 0. 执行摘要

此前对 PrimeAtlas 的理解存在两项关键偏差：

1. 把调性、行动、冲突、画像误解成四个并列的产品中心；
2. 把下一版本规划成“首页仪表盘 + 调性切换 + 习惯 CRUD”。

重新核对蓝图和 v6 原型后，准确结论是：

> **PrimeAtlas 是以“成为谁”为北极星的个人成长操作系统。用户定义当前身份和目标身份，系统把身份差距转化为目标、里程碑和专业成长计划，通过多 Agent 评审、交叉融合、场景自适应和执行反馈持续调整，最终让成果反过来更新身份认知和下一阶段路径。**

产品主循环是：

> **身份定义 → 目标推导与里程碑 → 交叉融合与多 Agent 决策 → 专业计划 → 每日执行与反馈 → 成果和数据洞察 → 身份/目标重校准**

v6 原型不是完整产品终态，而是以“单手扣篮”为高密度体能锚点的纵向体验样板。它已经较强地表达了目标量化、专业训练、身体数据、Prime Pulse、RPE、自适应调整、知识注入和 Agent 决策，但身份迁移、多域同构、统一时间编排和长期画像演进仍未形成完整体验。

因此，现有 `deliverables/s1/*` 不能作为下一版本的直接开发依据，必须重构。

## 1. 我之前理解偏在哪里

### 1.1 错把支撑机制当作顶层主轴

调性、冲突和画像很重要，但它们的正确位置分别是：

- **调性**：覆盖内容、视觉、交互、触感和推送的全层人格适配器；
- **冲突**：目标、时间、能量和安全约束进入计划编排时的决策能力；
- **画像**：记录身份变化、激活维度和过渡态叙事的演进载体；
- **执行**：计划落地和数据回流环节，不是孤立习惯打卡。

真正的顶层主轴只能是“用户要成为谁，以及系统如何持续帮助他接近这个身份”。

### 1.2 错把产品降维成自我管理/习惯陪伴 App

旧 `PRD-S1.md` 将产品描述为“AI 自我管理 / 成长陪伴 App”，并把习惯 CRUD、连续记录和今日焦点列为 Must。这会使产品直接落入 Fabulous、Finch、Habitica 一类习惯产品的比较框架，与蓝图“不是管理任务，是管理你成为谁”冲突。

### 1.3 错把旧 S1 当成蓝图 Sprint 1

蓝图 Sprint 1 明确定义为：

- A1 身份定义；
- A2 标签管理；
- B1 身份推导目标；
- B2 目标拆解里程碑。

它不是首页、调性和习惯 CRUD。旧 S1 使用了相同版本名，却承载了另一套范围，造成产品、架构、设计和 QA 全链路错位。

### 1.4 对“不阻断”的表述不够精确

正确边界是：

- 普通目标冲突和偏好冲突：权衡而非禁止，必须保留“一键采纳 / 我自己来”；
- 危险的具体执行方案：`SAFETY-OFFICER` 或 `REHAB-SPECIALIST` 有权否决；
- 安全否决必须给出证据、原因和降载/替代方案；
- 系统否决的是危险方案，不是用户拥有目标和查看信息的权利。

## 2. 修正后的产品定义

### 2.1 一句话定位

**PrimeAtlas — 你的个人成长操作系统。定义“我是谁”，系统为你生成并持续适配完整成长路径。**

### 2.2 用户真正雇佣它完成的工作

用户表面上需要的是健身计划、英语素材、阅读管理或每日安排；真正的问题是：

> 一个拥有多重角色、处于身份转型期的人，无法独立把多个成长方向、专业知识、有限时间、身体状态和不断变化的现实生活组织成一条连续、可信、可执行且不会因中断而失败的路径。

PrimeAtlas 的核心 Job 是：

1. 理解用户当前是谁、想成为谁；
2. 识别身份能力差距；
3. 推导目标和里程碑；
4. 通过专业 Agent 生成并评审计划；
5. 在时间、能量、场景和安全约束下统一编排；
6. 从执行、恢复、疼痛、RPE、学习和成果中持续学习；
7. 让成果反向更新身份叙事和下一阶段路径。

### 2.3 产品北极星叙事

- 不是“今天做了吗”，而是“离目标近了吗”；
- 不是“连续坚持了几天”，而是“哪些能力和身份证据正在形成”；
- 不是“计划有没有照表完成”，而是“现实变化后成长是否仍能持续”；
- 不是“AI 给了一个答案”，而是“专业决策是否可解释、可追溯、可调整”。

## 3. 完整产品飞轮

### 3.1 日循环

1. 用户通过 Prime Pulse 进入今日成长状态；
2. 系统展示今日计划、目标关联和调整原因；
3. 用户标记能量并执行训练/学习/恢复活动；
4. 记录完成质量、RPE、疼痛、状态和成果；
5. 系统更新负荷、能力、风险和计划参数；
6. 给出“这一步如何让你更接近目标身份”的反馈。

### 3.2 周期循环

1. 多 Agent 复盘执行、恢复、场景和成果；
2. 检测突破、里程碑、觉醒和节奏型 PR；
3. 发现用户未意识到的隐藏进步；
4. 调整周计划、时间槽和强度曲线；
5. 生成下一周期承诺；
6. 在有真实变化且用户确认时，创建新的画像版本和身份叙事。

### 3.3 重大变化循环

换工作、受伤、出差、家庭变化或兴趣变化发生时：

- 评估影响；
- 暂停、归档、恢复或重排目标；
- 保留历史记录；
- 重新运行冲突、安全和时间编排；
- 由用户确认身份与目标层面的变化。

## 4. 蓝图功能版图的正确关系

### 4.1 业务主干

- **A 身份系统**：当前身份、目标身份、身份标签、身份进度、过渡态；
- **B 目标系统**：目标推导、里程碑、仪表盘、重校准；
- **K 交叉融合引擎**：内容交叉、统一时间、场景自适应、知识流动、素材回流；
- **C 计划编排**：计划生成、时间槽、冲突调解、专业质量标准；
- **D 执行系统**：今日视图、执行追踪、即时反馈、微调；
- **E 成果系统**：PR 检测、庆祝、成长回顾和选择性分享；
- **I 数据洞察**：数据中心、隐藏进步、趋势和健康报告。

### 4.2 状态与仪式层

- **F 能量系统**：主观能量、睡眠、HRV、恢复和强度调整；
- **G Prime Pulse**：每日启动、目标承诺、势能火焰和休息日仪式；
- **H 调性系统**：全层人格与表达适配。

### 4.3 基础保障层

- **J 基础架构**：账号、内容、通知、隐私、离线和同步。

### 4.4 多 Agent 专业决策层

12 个 Agent 不是 12 个聊天机器人，而是不同职责和权限的专业决策网络：

- 领域 Agent 负责专业方案；
- `TIME-ORCHESTRATOR` 负责跨目标时间编排；
- `TONE-ADAPTER` 负责一致表达；
- `USER-PROFILE` 维护画像输入；
- `SAFETY-OFFICER` 与 `REHAB-SPECIALIST` 拥有安全否决权。

用户可以展开依据、采纳建议或主动覆盖普通建议，但危险方案不能被系统直接输出。

## 5. v6 原型实际表达了什么

### 5.1 六个导航不是六个平级工具

- **今日**：身份目标、身体状态、能量、计划和调整原因的总控台；
- **身体**：身体数字孪生、准备度、ACWR 和目标指标的数据模型；
- **训练**：完整专业计划的执行台；
- **目标**：目标、差距、路线和里程碑；
- **知识**：知识输入、RAG、领域术语、知识注入和素材回流；
- **我的**：身份信息、周期、调性、数据和隐私设置。

### 5.2 应直接继承的交互真源

- `openAIGoalChat()`：五阶段 AI 目标画像构建的交互方向；
- `startPulseHold()` / `handlePulseTap()`：Prime Pulse 长按启动仪式；
- `completeAction()`：动作完成后进入 RPE 反馈；
- `showAdjustDetail()`：展开查看 Agent 协作和自动调整理由；
- `showKnowledgeInjection()`：知识进入计划前的评估与用户确认；
- `showPRCelebration()`：成果驱动的庆祝体验；
- `switchTone()` / `selectTone()` / `applyToneToContent()`：调性选择和全局表达适配方向；
- 目标差距、训练阶段、动作视频、准备度和 ACWR 的白盒表达。

### 5.3 只是演示壳、不能照搬为产品规则的内容

- “单手扣篮”及固定训练数据只是体能锚点示例；
- 大量静态身体指标和固定百分比尚无真实数据模型支撑；
- Agent 卡片主要是可见性演示，尚未形成完整编排运行时；
- 六 Tab 信息密度和底部导航数量需在 Flutter 版重新验证；
- 原型中 emoji 功能图标、部分即时提示和硬编码视觉不能直接迁移；
- “今日进展百分比”若成为压力指标会违反成果驱动原则；
- Prime Pulse 当前同时承载长期目标势能、今日准备度和训练解锁，语义需要拆清。

### 5.4 原型与蓝图的关系

- 蓝图回答“完整产品为什么存在、系统如何工作”；
- 原型回答“体能锚点用户每天如何感知和操作其中一条纵向闭环”；
- 开发时应保留原型已验证的交互品质，但必须把蓝图缺失的身份、多域融合和长期演进补进去；
- 不能逐像素翻译原型，也不能只按抽象蓝图开发没有真实体验的后台系统。

## 6. 体能锚点与成长 OS 的关系

体能是首个深扎域，原因是：

- 目标可量化；
- 计划专业性可被感知；
- 执行与反馈闭环清晰；
- 安全、恢复、RPE 和 ACWR 能证明系统不是普通待办工具。

但体能不能成为产品身份。正确战略是：

> **单域深扎 + 融合可见。**

即：体能方案做到足够专业，同时让派生语言、知识注入、素材回流和身份目标在训练主流程中自然可见，证明它是成长 OS，而不是在健身 App 外面加几个英语和阅读入口。

## 7. 用户主权与安全边界

### 7.1 普通冲突

- 系统提供权衡建议；
- 透明展示依据；
- 保留“一键采纳 / 我自己来”；
- 不羞辱、不惩罚、不制造失败感；
- 用户可继续保留目标和自行编排。

### 7.2 安全冲突

- 安全/康复 Agent 可以否决危险的具体方案；
- 必须展示可追溯原因；
- 必须提供降载、替代、暂缓或重新评估条件；
- 不因否决方案而否定用户目标；
- 涉及身体敏感数据时需明确 consent，权限拒绝后优雅降级。

### 7.3 需要修正文档表述的地方

既有三缺口 PRD 中“身体安全通道也不阻断”的表达过宽。应改为：

> 普通冲突的 `blocked_user` 恒为 false；安全通道可以阻止系统生成或执行危险方案，但不能阻断用户管理目标、查看依据和选择安全替代方案。

## 8. 产品不可妥协红线

1. 不以连续天数、排行榜、红点和罪感驱动用户；
2. 所有反馈以目标邻近度、成果和身份证据为中心；
3. 不因计划中断把用户判定为失败；
4. 计划必须适应加班、出差、生病、假期和能量变化；
5. 多域共享目标—计划—执行—成果的心理和数据框架；
6. 专业计划必须经过领域评审和安全审核；
7. 体能主训练必须包含热身、主训练、拉伸和合理动作顺序；
8. 同一时间槽只执行一个领域，不做边练边背、组间学习等伪融合；
9. 知识进入实践必须经过提取、评估、用户确认和渐进融入；
10. 未激活维度不渲染、不占存储、不进入权重；
11. 系统不能静默改写用户画像；
12. 调性必须同会话一致，严厉也不得羞辱或制造焦虑；
13. 成长默认私密，分享必须主动；
14. 核心执行、今日视图和能量记录应离线可用；
15. 系统不越界进行医疗诊断或心理评估。

## 9. 旧 S1 文档如何处理

### 9.1 `PRD-S1.md`

**结论：业务范围废弃，少量基础需求迁移。**

废弃：习惯 CRUD、连续记录、陪伴微步作为下一版本主轴，以及“AI 自我管理/成长陪伴 App”定位。

迁移：Onboarding 的 consent、数据归属、离线优先、调性入口、空/错/权限状态设计。

### 9.2 `ARCH-S1.md`

**结论：技术骨架可复用，业务模块和 API 推断必须重写。**

可保留：

- `lib/core` / `lib/app` 单向依赖；
- Riverpod、GoRouter、Repository、Provider、Theme 分层；
- SQLite 本地优先与 `SyncAdapter` 扩展；
- ADR-6 snake_case；
- 架构守护测试；
- 调性语义 Token 单一真源。

必须重写：

- 页面目录和路由；
- goals/habits/tasks 表结构；
- 根据契约“推断”的方法签名；
- 事件时间精度说明；
- 把冲突、画像仅做占位的安排；
- S1 只完整落调性的范围定义。

### 9.3 `UIUX-S1.md`

**结论：设计系统局部保留，信息架构重做。**

可保留：语义 Token、Lucide、可访问性、克制动效、四调性映射、深浅色能力。

必须删除或重做：连续火焰、习惯详情、今日微步、泛陪伴式首页。新首页必须先表达“我正在成为谁、离目标多远、今日计划为何如此、哪些融合正在发生”。

### 9.4 `QA-S1.md`

**结论：测试方法保留，测试对象重写。**

保留：core/unit/widget/integration/golden 分层和红线门禁。

重写：习惯 CRUD、连续记录和旧首页用例；新增身份定义、目标推导、里程碑、融合证据链、专业计划安全、用户覆盖和安全否决测试。

## 10. 对开发路线的直接影响

### 10.1 不能做的事

- 不能继续按旧 S1 直接开发首页、习惯 CRUD 和设置；
- 不能把原型页面逐像素翻译成 Flutter；
- 不能机械照搬蓝图原始 Sprint，把已经冻结和实现的 S0 核心能力重新做一遍；
- 不能把三大硬缺口专项与完整蓝图割裂成两个产品；
- 不能继续用同一个“S1”名称承载多套不同范围。

### 10.2 应采用的路线原则

1. 远程仓库和现有 S0 Core 先恢复到干净、可追溯状态；
2. 下一小版本必须建立新的版本名和 Spec，避免与旧 S1 混淆；
3. 先补身份—目标主干，让产品第一次真正回答“我是谁、要成为谁”；
4. 随后接体能锚点计划和执行闭环；
5. 融合可见机制必须长在主流程中，而不是二级入口；
6. 调性、冲突、画像红线随相关功能逐步接入，不再做脱离业务的孤立模块；
7. 每个版本都必须是可体验的完整子集，并经过 analyze、test、QA、提交、远程验证。

## 11. 建议的近期版本序列

> 最终编号待架构文档收口后锁定；以下是产品依赖顺序，不能倒置。

### V0.2：身份与目标地基

- 身份定义五阶段流程；
- 当前身份/目标身份确认；
- 活跃维度与 consent；
- 目标推导与用户确认；
- 里程碑拆解和目标路线；
- 专业默认调性；
- 本地持久化和事件记录。

**验收**：用户能从自由表达愿望走到一组确认后的目标和里程碑；没有习惯打卡、连续天数和静默画像改写。

### V0.3：体能锚点专业计划

- 身体基础数据渐进录入；
- 单手扣篮示例目标映射；
- 日/周计划生成；
- 热身—主训练—拉伸强制结构；
- 时间槽、能量和基础场景参数；
- 安全审查与理由卡。

**验收**：计划结构专业、可解释；危险方案不输出；数据不足时保守降级。

### V0.4：今日执行与 Prime Pulse

- 今日总控台；
- Prime Pulse 长按启动；
- 能量标记；
- 场景化训练卡；
- 动作详情/视频入口；
- 滑动完成、RPE 和疼痛反馈；
- 调整理由展开；
- 离线执行。

**验收**：完整走通“查看原因→启动→执行→反馈→计划参数回流”，不以进度百分比制造压力。

### V0.5：融合可见 V1

- 训练卡内派生领域术语；
- 派生源溯源；
- 常驻知识注入入口；
- 专家门控与用户逐条确认；
- 今日融合证据链；
- 训练素材回流知识库。

**验收**：用户无需进入独立英语课程即可感知跨域成长；知识从不静默进入计划；融合没有红点、打卡和排名。

- 多目标时间/能量冲突；
- 一键采纳 / 我自己来；

**验收**：普通冲突不剥夺用户选择；安全风险阻止危险方案；未激活维度完全消失；画像变化必须有 consent。

## 13. 最终判断

PrimeAtlas 的完整价值可以概括为四句话：

- **Prime Pulse 让用户愿意开始；**
- **多 Agent 与专业计划让用户敢于相信；**
- **交叉融合与场景自适应让成长能够持续；**
- **PR、隐藏进步和身份叙事让用户长期留下。**

但这四项仍需围绕同一个核心服务：

> **帮助用户从“我现在是谁”，持续走向“我想成为谁”。**

后续所有需求、设计、架构、代码和测试，都必须能回答它在这条身份迁移路径中的位置；回答不了的功能，不应进入主版本范围。
