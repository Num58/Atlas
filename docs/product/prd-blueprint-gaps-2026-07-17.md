## 1. 产品目标（3 个清晰、正交的目标）

1. **调性千人千面且全层一致**——用一套 CSS 语义变量引擎让 4 种调性（热血/专业/陪伴/严厉）在文案、视觉、交互、推送全层一致生效，强化身份认同，规避"健身 App"归类。
2. **冲突以权衡而非禁止保护自主权**——冲突检测呈"权衡提示 + 透明裁决双轨（一键采纳 / 我自己来）"，建立"专业可靠"信任根基，绝不硬阻断用户。
3. **成长可见可讲述**——画像动态更新以"过渡态叙事 + 动态轴雷达 + 版本化"让身份进度飞轮闭环，未用维度不占位、系统不自动改画像。

> 三者正交：目标 1 解决"语气一致性"，目标 2 解决"自主权保护"，目标 3 解决"成长显影"，共同服务于五条铁律（尤其"身份认同>意志力""专业可靠是信任根基"）。

## 2. 用户故事（5 个场景）

1. **首次进入·主调锚定**：小琳首次打开 PrimeAtlas，系统据其目标推荐"热血"主调，全程文案/视觉/推送语气一致；她可在设置里切到"专业"，但同会话内不再漂移。
2. **冲突目标·权衡提示**：阿杰同时设"增肌"与"减脂"，系统轻提示二者存在权衡，给出推荐编排并透明展示依据；他点"一键采纳"或切"我自己来"手动调滑块。
3. **身体信号·安全否决**：怀孕期的用户录入生理状态，系统在"负荷 vs 恢复"冲突中走身体安全通道，给出可追溯理由且不阻断其查看原计划。
4. **坚持三月·身份过渡**：坚持 3 个月后，用户画像从"新手"过渡到"进阶"，雷达图长出新轴（带生长动画），过渡态叙事条讲述"你已从 X 成为 Y"，严厉型措辞用"陪你成为"无焦虑。
5. **想自己调·覆盖权保留**：凡系统给出推荐，用户始终能看到"我自己来"入口，覆盖后系统记录其显式确认来源，绝不静默改写画像。

## 3. 用户研究洞察（来自瑞思）

- **克制化诉求**：用户普遍反感"被催促/被评判"，与五条铁律"不加罪、不阻断"高度一致；严厉型调性必须以"陪你成为"呈现，量表不应触发焦虑。
- **归类焦虑**：用户一旦感知"这是又一个健身 App"即流失意愿上升；调性系统（尤其是文案层与推送层）是脱离归类感知的第一触点。
- **自主权敏感**：冲突场景用户接受"提示"但拒绝"禁止"；保留"我自己来"是信任前置条件（弃用主因=感觉被替做决定）。
- **成长可见性**：用户需要"我变了"的可讲述证据，而非又一张分数卡；过渡态叙事 + 动态雷达满足"身份进度飞轮"的情感闭环。
- **未用维度不占位**：雷达仅显活跃维度，用户初期可能困惑"其他维度在哪"，需用"下一阶身份锚定"过渡态 + onboarding 渐进披露化解。

## 4. 竞品对比（来自竞析，市场定位对齐）

> 竞析维度：健身/习惯类 App（Keep、Zepp、Apple Fitness+、Fabulous、Coach.ai、Notion 习惯模板）在"调性/冲突/画像"三缺口的覆盖度普遍薄弱。

| 功能维度 | 我方（补完后） | 健身类 App | 习惯类 App | 通用效率工具 |
|---------|--------------|-----------|-----------|------------|
| 全层调性适配 | ✅ 4 调性 CSS 语义变量全层一致 | ❌ 单语气 | ⚠️ 有限 | ❌ 无 |
| 冲突检测·权衡非禁止 | ✅ 双轨裁决 + 安全通道 | ❌ 无/硬阻断 | ⚠️ 简单提醒 | ❌ 无 |
| 画像动态轴雷达 | ✅ 仅活跃维度 + 过渡叙事 | ⚠️ 固定维度 | ❌ 分数卡 | ⚠️ 自定义但静止 |
| 北极星动态权重 | ✅ 未用维度权重=0 | ❌ 固定 MAU 逻辑 | ❌ | ⚠️ 部分 |
| 归类规避 | ✅ 身份操作系统定位 | ❌ 强健身归类 | ⚠️ | ✅ 但无成长内核 |

**竞析对齐点（已并入排期）**：
- M1 默认**单主调 + 渐进解锁**（Phase1 锁主调），推送 + 文案层先行，避免"换皮感"。
- M2 冲突呈**权衡非禁止**、约束 B 显性化为"聚焦"卖点（C6）。
- M3 雷达用**身份角色命名**、过渡态用**下一阶身份锚定**，规避"其他维度在哪"困惑。

## 5. 数据依据（来自数析）

- **指标口径（数析确认）**：调性 / 冲突 / 画像三模块指标均为 **Process / Guardrail** 类型，**不进北极星**（身份进度指数动态权重不变）。
- **埋点先行**：M0 完成 `identity_event_bus` + 埋点 schema（tone_change_event、content_tone_tag、goal/conflict/arbitration、profile_field_update、identity_transition_event、dimension_data_presence），作为 M1–M3 红线验收的**数据前置**；数析负责契约 sign-off，避免"无信号可验"。

## 7. 关键流程图 / UI 设计稿

### 7.1 调性切换流（同会话守恒）
```
[进入会话] → 读取 content_tone_tag(主调) → 全层应用 --tone-* 根变量
   └─ 同会话内任何触点复用同一 tag（T-RL2）
[用户主动切调] → tone_change_event 记录 → 仅跨会话渐进解锁新调（T-RL1 健康带宽）
```
> 实现要点：T2 引擎以 CSS 语义变量（`--tone-primary / --tone-radius / --tone-font / --tone-ease`）切根，单一真源，非多套 UI（防换皮）。设计稿基于 `primeatlas-prototype-v6.html` 调性中心扩展。

### 7.2 冲突裁决双轨
```
[目标写入] → C1 引擎扫描冲突类型 → [命中]
   ├─ 身体类 → C4 安全通道：给可追溯理由，不阻断（C-RL3）
   └─ 普通类 → C2 轻提示卡（权衡非禁止）
        ├─ 一键采纳 → 透明展示改动依据（C-RL2）
        └─ "我自己来" → 高级仲裁滑块，保留覆盖入口（C-RL2）
```

### 7.3 动态轴雷达 + 过渡态叙事
```
[identity_transition_event 有可讲述变化?]
   ├─ 否 → 叙事条不显（P-RL3）
   └─ 是 → 叙事条讲述"从 X 到 Y"（下一阶身份锚定）
[dimension_data_presence] → 仅渲染已激活维度（P-RL1），新轴生长动画
[profile_field_update] → 须带"用户确认来源"，自动源占比=0（P-RL2）
```

## 8. Non-goals（明确不做什么）

- ❌ **不自动改写用户画像**：任何画像变更需用户显式确认（P-RL2）。
- ❌ **不以禁止/阻断方式处理冲突**：冲突一律呈权衡提示（C-RL1）。
- ❌ **不为未激活维度保留占位卡/灰块**：未用维度完全不出现（P-RL1）。
- ❌ **不做多套独立 UI 换皮**：调性统一用 CSS 语义变量引擎，单一真源（防换皮、防漂移）。
- ❌ **三大模块指标不进北极星**：均为 Process/Guardrail，身份进度指数动态权重不变。
- ❌ **不在 Phase1 引入社交/排行榜/好友机制**：避免归类回健身 App 与焦虑源。
- ⚠️ **C4 身体冲突安全通道 Phase1 即接入（用户拍板 2026-07-17）**：需显式 consent + 合规；权限拒绝时优雅降级，不阻断核心（非默认强制开启）。
- ❌ **不在同会话内漂移调性**：同会话守恒（T-RL2）。

### 9.3 验收红线正式版（9 条，附判定）

**调性（T）**
- **T-RL1 切换健康带宽**：合格 = 周切换 1–2 次且系统不阻拦；不合格 = 阻止切换或将切换标记为罪感。
- **T-RL2 同会话守恒**：合格 = 同会话调性一致、跨会话渐进解锁；不合格 = 会话内无故漂移。
- **T-RL3 严厉不焦虑**：合格 = 严厉主调无罪感/威胁/羞辱词，呈"陪你成为"；不合格 = 焦虑触发词或量表显著升。

**冲突（C）**
- **C-RL1 不硬阻断**：合格 = 冲突呈权衡/提示可继续；不合格 = 禁止/阻断拦截。
- **C-RL2 透明 + 保留覆盖**：合格 = 裁决双轨透明展示依据且保留"我自己来"；不合格 = 强制默认、无覆盖入口。
- **C-RL3 身体理由可查**：合格 = 身体冲突给可追溯理由；不合格 = 理由缺失。

**画像（P）**
- **P-RL1 未用维度不占位**：合格 = 未激活维度不占界面/存储；不合格 = 保留占位卡/灰块。
- **P-RL2 不自动改画像**：合格 = 变更需用户确认；不合格 = 系统自动改写。
- **P-RL3 叙事按变化显隐**：合格 = 仅 `identity_transition_event` 有可讲述变化才显；不合格 = 无变化也提示。
