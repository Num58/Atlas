## 第一部分：产品定义

### 1.1 产品一句话定位

**PrimeAtlas — 你的个人成长操作系统。定义"我是谁"，系统为你生成并持续适配完整成长路径。**

### 1.2 核心价值主张

| 维度 | 传统产品 | PrimeAtlas |
|------|---------|-----------|
| 驱动模式 | 打卡驱动（今天做了吗） | 目标驱动（离目标近了吗） |
| 身份认知 | 用户 = 任务执行者 | 用户 = 自己定义的身份拥有者 |
| 模块关系 | 独立App或独立模块 | 统一框架下的交叉融合 |
| 反馈机制 | 压力型（连续打卡/排行榜焦虑） | 成果型（PR庆祝/隐藏进步发现） |
| 计划适配 | 固定计划，断了就失败 | 自适应计划，永不断档 |
| 调性 | 千篇一律的产品性格 | 四种调性自由切换 |
| 用户画像 | 单一标签用户 | 多标签交叉立体人 |

**一句话价值主张**：不是管理你的任务，是管理你成为谁。

### 1.3 产品设计铁律

#### 铁律一：目标驱动 > 打卡驱动

- **原则**：系统不问"你今天做了吗"，而是问"你离目标近了吗"
- **实现**：所有交互入口以目标进度为核心，不暴露连续打卡天数
- **反面**：不出现"连续X天"的主视觉展示；不因断签惩罚用户

#### 铁律二：成果驱动 > 压力驱动

- **原则**：正面反馈替代焦虑，PR庆祝、隐藏进步发现
- **实现**：系统主动发现用户没意识到的进步，主动触发庆祝
- **反面**：不使用排行榜制造焦虑；不使用红点制造强迫点击

#### 铁律三：身份认同 > 意志力消耗

- **原则**：让用户认同"我就是这样的人"，行为自然发生
- **实现**：目标绑定身份标签，系统语言强化身份认同
- **反面**：不使用"坚持就是胜利"等消耗意志力的文案

#### 铁律四：自适应 > 固定计划

- **原则**：加班自动转方案，出差自动换计划，永不断档
- **实现**：场景感知 + 计划弹性编排引擎
- **反面**：不要求用户手动调整计划；不断签惩罚

#### 铁律五：心理层统一

- **原则**：健身/英语/读书共享同一套目标-执行-成果框架
- **实现**：统一的数据模型、统一的进度语言、统一的反馈体系
- **反面**：不为不同领域设计不同的追踪方式

#### 铁律六：专业可靠是信任根基 ⭐

- **原则**：每一份计划都必须专业到用户可以无条件信赖
- **实现**：多Agent协作设计+交叉评审，确保计划的每个细节都经得起专业推敲
- **涵盖范围**：
  - 训练容量计算（总容量、局部容量、ACWR）
  - 动作间衔接逻辑（爆发力→力量→辅助→拉伸的顺序不可颠倒）
  - 运动前热身（动态热身、神经激活、关节活动度）
  - 运动后拉伸（静态拉伸、筋膜放松、呼吸调节）
  - 晨间微训练（居家激活、关节唤醒、低强度）
  - 午间恢复（工位拉伸、呼吸冥想、姿态调整）
  - 晚间放松（深层拉伸、睡眠准备、神经放松）
  - 英语学习难度阶梯（i+1可理解输入，不跳跃）
  - 读书进度与理解深度平衡（不追求速度追求转化）
- **反面**：
  - 不允许生成"随便练练"的敷衍方案
  - 不允许只给动作不给热身和拉伸
  - 不允许不考虑动作间逻辑衔接的随机组合
  - 不允许不经过安全Agent审查的计划直接输出
  - 不允许在组间歇做任何分散注意力的事（背单词、看视频等）
- **用户感知**：
  - "这个计划太专业了，连热身和拉伸都帮我安排好了"
  - "晨间、午间、晚间都有针对性的安排，很贴心"
  - "我知道为什么先做这个动作再做那个动作，逻辑很清晰"
  - → **信任就是这样建立的**

### 1.4 非目标（明确不做什么）

| 类别 | 不做的内容 | 原因 |
|------|-----------|------|
| 社交 | 公域内容流 / 社区 / 动态 | 成长是私密的，不需要公域社交 |
| 内容 | 自建课程体系 / 教练平台 | 不做内容生产者，做成长操作系统 |
| 电商 | 装备商城 / 知识付费 | 不做交易平台 |
| 打卡 | 连续打卡排行榜 / 打卡群 | 与目标驱动理念冲突 |
| 健康 | 医疗诊断 / 心理健康评估 | 不做医疗级产品 |
| AI | AI教练对话（初期不做） | 先做确定性规则引擎，AI逐步引入 |
| 反人类 | 组间歇练口语 / 边跑边背单词 | 违反交叉融合原则 |

### 1.5 与竞品的本质差异

| 竞品类别 | 代表产品 | 核心差异 |
|---------|---------|---------|
| 健身App | Keep / 训记 | PrimeAtlas不是健身App，健身只是成长维度之一 |
| 打卡App | 小习惯 / Habitica | PrimeAtlas不追求连续打卡，追求目标进度 |
| 英语App | 多邻国 / 百词斩 | PrimeAtlas的英语内容来源于用户真实成长场景 |
| 读书App | 微信读书 / 得到 | PrimeAtlas的阅读是知识输入，融入成长闭环 |
| 目标管理 | 飞书OKR / Notion | PrimeAtlas不做通用目标管理，专注个人成长领域 |

**本质差异**：所有竞品解决"怎么做"的问题，PrimeAtlas解决"成为谁"的问题。

## 第二部分：用户体系

### 2.1 多标签交叉身份模型

#### 核心理念

用户不是单一标签，是多标签交叉的立体人。系统不给用户贴标签，用户自己定义"我是谁"和"我想成为什么样的人"。

#### 标签体系结构

```
用户身份 = {
  当前身份标签集合: ["程序员", "篮球爱好者", "英语初学者"],
  目标身份标签集合: ["国际篮球教练", "流利英语沟通者", "体能训练专家"],
  过渡态描述: "从程序员转型为国际篮球教练的路上"
}
```

### 2.2 核心用户完整故事

#### 用户A：从程序员到国际篮球教练

**身份定义**
- 我是：程序员 + 篮球爱好者 + 英语学习者
- 我想成为：国际篮球教练 + 流利英语沟通者 + 体能训练专家

**成长故事线**

```
阶段1：建立基础（第1-3个月）
  - 体能：建立基础体能框架，每日45分钟训练
  - 英语：掌握篮球基础术语200词，每日20分钟
  - 读书：阅读《篮球战术体系》+《运动训练学》，每周3次

阶段2：交叉融合（第3-6个月）
  - 交叉：训练中融入战术分析（体能→篮球知识）
  - 交叉：阅读的战术体系用英语做笔记（读书→英语）
  - 交叉：英语学习内容切换为NBA战术解说（英语→篮球）
  - 知识流动：运动训练学理论优化个人训练计划

阶段3：实践输出（第6-12个月）
  - 输出：用英语撰写训练计划
  - 输出：模拟战术布置（英语表达）
  - PR：拿到教练资格证
  - PR：第一次用英语完成战术布置

阶段4：身份确认（第12个月+）
  - 系统反馈：系统确认身份迁移进度
  - 新的目标：带领第一支队伍
  - 标签更新：从"转型中"到"国际篮球教练"
```

**关键设计点**
- 英语学习内容是篮球术语/战术沟通（内容交叉，不是同时做）
- 读的体能书中的方法融入训练计划（知识流动）
- 训练内容自动生成英语学习素材（素材回流）
- 不做：组间歇练口语、边跑边背单词

#### 用户B：恢复羽毛球 + 职场回归的外企宝妈

**身份定义**
- 我是：新手妈妈 + 外企职员（产假中） + 前羽毛球爱好者
- 我想成为：职场回归者 + 羽毛球回归者 + 精力管理达人

**成长故事线**

```
阶段1：恢复期（第1-3个月）
  - 体能：产后恢复训练，每日20分钟低强度
  - 英语：职场英语恢复，每日15分钟听力
  - 读书：精力管理/时间管理，每周2次
  - 自适应：婴儿作息不稳定，系统自动适应碎片化时间

阶段2：重建期（第3-6个月）
  - 体能：羽毛球基础体能恢复，每日30分钟
  - 英语：商务英语场景训练（会议/邮件），每日20分钟
  - 读书：行业知识更新，每周3次
  - 交叉：英语听力素材切换为羽毛球赛事英文解说
  - 自适应：产假结束过渡期，时间自动调整

阶段3：回归期（第6-9个月）
  - 交叉：商务英语能力支撑职场回归
  - 交叉：精力管理理论优化训练安排
  - PR：完成第一场回归羽毛球赛
  - PR：职场回归第一周顺利完成

阶段4：新常态（第9个月+）
  - 身份确认：不再是"回归者"，是"掌控者"
  - 标签更新：职场人 + 羽毛球手 + 精力管理者
```

**关键设计点**
- 场景自适应：宝宝睡觉时间 = 训练窗口，系统自动编排
- 碎片化学习：英语和阅读支持5-15分钟短单元
- 恢复追踪：不追求增量，追求恢复曲线
- 调性适配：陪伴型调性，理解和鼓励

### 2.3 AI引导式目标画像构建系统 ⭐

> **核心设计原则**：没有预设标签库。用户自由输入，AI像教练一样层层引导，逐步精准定位。

#### 2.3.1 设计理念

```
传统做法（错误）：预置标签库 → 用户从库里选 → 像填表
PrimeAtlas（正确）：用户自由表达 → AI引导追问 → 像和教练对话
```

**为什么不用标签库**：
- 标签库是"我们觉得用户是谁"，不是"用户觉得我是谁"
- 真实用户是立体的、独特的，无法被预设标签穷举
- 填表体验和成长产品的调性背道而驰

#### 2.3.2 五阶段引导对话流程

```
阶段1：种子收集        阶段2：维度展开        阶段3：关联发现
"你想提升什么？"        "篮球的哪个方面？"      "你有没有想过..."
用户自由输入           AI按维度追问           AI主动建议关联方向
    ↓                      ↓                      ↓
阶段4：深度追问        阶段5：画像合成
"到什么程度？"         生成精准目标画像
"有什么限制？"         用户确认/调整
```

**阶段1：种子收集**
- 用户自由输入任何内容（词、短语、句子）
- 示例："篮球" / "我想扣篮" / "产后恢复加学英语"
- AI不做任何限制，不引导，纯收集

**阶段2：维度展开**
- AI根据用户输入，按维度追问
- 篮球 → "你关注篮球的哪个方面？体能？技巧？战术？身体素质？"
- 用户选择/输入后继续："身体素质方面，具体是？对抗？突破？弹跳？协调性？"
- 用户也可以手动输入AI没提到的方向

**阶段3：关联发现**
- AI主动建议用户可能没想到的方向
- "你选了篮球和弹跳——踝关节的灵活性是否需要关注？"
- "你的程序员背景在战术分析中可能很有用，要纳入目标吗？"
- 用户可以选择采纳或跳过

**阶段4：深度追问**
- 确定目标精度：当前水平？期望达到什么程度？有时间限制吗？
- 确定约束条件：可用时间？场地限制？身体限制？
- "你希望6个月内扣篮，还是稳步提升没有时间压力？"

**阶段5：画像合成**
- AI生成结构化目标画像
- 展示给用户确认，支持修改任何部分
- 确认后进入目标承诺仪式

#### 2.3.3 AI追问策略

```
追问原则：
1. 渐进式：从宽到窄，从大到小
2. 相关性：只追问与用户已输入相关的内容
3. 可选性：所有追问都可以跳过
4. 尊重性：用户说"够了"就停止追问
5. 惊喜感：主动发现用户没意识到的关联

追问轮次：建议3-6轮，用户可以随时终止
追问语气：根据用户选择的调性适配
  热血型："来，告诉我你的野心！"
  专业型："请描述你的具体目标领域"
  陪伴型："慢慢来，想到什么就说什么"
  严厉型："说清楚，你到底要什么"
```

#### 2.3.4 意图识别机制

用户输入可能是任何形式，AI需要识别并结构化：

| 输入示例 | AI识别 | 追问方向 |
|---------|--------|---------|
| "篮球" | 运动专项→篮球 | 体能/技巧/战术/身体素质？ |
| "我想扣篮" | 运动专项→篮球→爆发力→弹跳 | 当前摸高？训练时间？ |
| "产后恢复" | 康复→产后→核心/盆底肌 | 顺产/剖腹产？多久了？ |
| "雅思7分" | 语言→英语→考试→雅思 | 听说读写哪个最弱？ |
| "能把书用起来" | 认知→读书→实践转化 | 什么类型的书？ |
| "篮球+英语+教练" | 多标签交叉 | 三者如何关联？英语执教？ |

**模糊输入处理**：
- "我想变得更好" → "很好，哪个方面？身体？技能？知识？语言？"
- "什么都想提升" → "那我们先从一个开始，你觉得最紧迫的是什么？"

#### 2.3.6 画像动态更新与目标切换

画像不是一成不变的，随用户成长和意愿而演进。

**核心能力：多目标并行**

- 用户可同时进行 **最多3个目标**
- 目标之间可以有关联（交叉融合），也可以完全独立
- 统一时间编排引擎对3个目标的时间进行统筹分配
- 每个目标有独立的进度追踪和PR庆祝

**目标切换机制**

```
用户主动切换：
  "我不想练篮球了，我想开始游泳"
  → AI引导对话（同首次创建流程）
  → 生成新目标画像
  → 旧目标归档（数据保留，可随时恢复）
  → 统一成长方案重新编排

用户增删目标：
  当前：篮球弹跳 + 英语学习
  新增：增肌减脂
  → AI检查：三个目标是否有冲突？
     "增肌和弹跳训练可以互补。但增肌需要热量盈余，如果同时想减脂会有冲突。"
  → 用户确认后，三个目标统一编排

目标暂停与恢复：
  "最近太忙，篮球先暂停"
  → 目标进入暂停状态
  → 时间重新分配给其他活跃目标
  → 随时可以恢复，历史数据完整保留
```

**目标生命周期状态**

| 状态 | 说明 | 用户操作 |
|------|------|---------|
| 🟢 活跃 | 正在进行中 | 日常执行 |
| 🟡 暂停 | 主动暂停，数据保留 | 随时恢复 |
| 🔵 调整中 | 正在修改目标参数 | 重新引导对话 |
| ⚪ 归档 | 已完成或放弃 | 可查看历史，可重新激活 |
| 🟣 新创建 | 刚创建，尚未开始执行 | 确认后激活 |

**画像更新触发条件**

| 触发条件 | 更新动作 |
|---------|---------|
| 用户主动切换目标方向 | 重新进入引导对话，旧目标归档 |
| 用户新增/删除目标 | 重新编排时间分配 |
| 完成阶段性目标 | AI建议更新目标层级或追加新目标 |
| 发现新兴趣方向 | 追加到画像中（不超过3个活跃目标） |
| 长期未涉足某领域 | AI询问是否暂停或归档 |
| PR达成 | 自动提升该维度水平等级 |
| 生活状态变化 | 用户可更新约束条件（时间/场地） |

**三个目标并行示例**

```
用户：转型国际篮球教练的程序员

目标1（主）：篮球弹跳训练 + 体能
  ├── 每周4次训练，总投入5h
  ├── 关联：训练素材回流为篮球英语内容
  └── 关联：体能书籍知识注入训练计划

目标2：篮球英语（教练方向）
  ├── 每周5次学习，总投入3h
  ├── 关联：内容来自目标1的训练素材
  └── 独立：基础语法/听力训练

目标3：读书实践（运动训练学）
  ├── 每周3次阅读，总投入2h
  └── 关联：知识注入目标1的训练计划

统一编排：
  总周投入10h，交叉内容自动合并（节省约1.5h）
  加班时优先保留交叉内容，砍独立训练
```

**目标冲突检测**

| 冲突类型 | 检测规则 | 处理方式 |
|---------|---------|---------|
| 方向冲突 | 增肌+减脂同时进行 | 提示冲突，建议分阶段或调整 |
| 时间冲突 | 三个目标总需求时间 > 可用时间 | 建议降低某目标频率或暂停 |
| 身体冲突 | 高强度训练+伤病康复同时 | 提示风险，强制调整强度 |
| 交叉度低 | 三个目标互不相关 | 提示但不阻止，用户自主决定 |

> 系统不强制阻止任何目标组合，但提供清晰的冲突提示和决策辅助。 |

### C6. 专业计划设计标准 ⭐ 信任根基

> 每一份计划必须专业到用户可以无条件信赖。这是Agent协作体系的核心产出质量标准。

#### C6.1 全天候场景覆盖

系统必须为用户的每个时间窗口提供专业方案，而非仅仅"健身房训练"：

| 时段 | 场景 | 方案类型 | 设计要点 |
|------|------|---------|---------|
| 🌅 晨间 | 居家/酒店 | 激活唤醒 | 动态拉伸、关节活动度、神经激活、低强度（5-15min） |
| 🌞 午间 | 工位/办公室 | 恢复舒展 | 姿态矫正、肩颈拉伸、呼吸冥想、微运动（5-10min） |
| 🌆 傍晚/晚间 | 健身房/户外 | 主训练块 | 完整训练流程（热身→主训练→拉伸），容量科学计算 |
| 🌙 睡前 | 居家 | 放松准备 | 深层拉伸、筋膜放松、神经放松、睡眠引导（5-10min） |

#### C6.2 单次训练专业标准

每个主训练块必须包含以下完整流程，不可缺失：

```
训练流程（强制性结构）：
┌─────────────────────────────────────────┐
│ 1. 热身阶段（不可跳过，不可合并到训练）   │
│    ├── 一般热身：5min 低强度有氧          │
│    ├── 动态拉伸：目标肌群动态活动度        │
│    ├── 神经激活：轻重量/快速动作预热       │
│    └── 专项准备：主训练动作的轻量预演      │
├─────────────────────────────────────────┤
│ 2. 主训练阶段                            │
│    ├── 爆发力/速度动作（如有）→ 最先      │
│    ├── 复合力量动作 → 其次               │
│    ├── 辅助/孤立动作 → 再次              │
│    ├── 核心/纠正性训练 → 最后             │
│    └── 动作间衔接逻辑：                   │
│        · 大肌群→小肌群                   │
│        · 多关节→单关节                   │
│        · 高神经需求→低神经需求           │
│        · 拮抗肌群可超级组配对             │
├─────────────────────────────────────────┤
│ 3. 拉伸放松阶段（不可跳过）              │
│    ├── 静态拉伸：本次训练目标肌群 30s×2   │
│    ├── 筋膜放松：泡沫轴/按摩球            │
│    └── 呼吸调节：2-3min 腹式呼吸          │
└─────────────────────────────────────────┘
```

#### C6.3 训练容量计算标准

所有训练计划的容量必须经过科学计算，不可凭经验随意编排：

| 容量指标 | 计算方式 | 安全阈值 | 监控频率 |
|---------|---------|---------|---------|
| 单次训练容量 | Σ(组数×次数×重量) | 不超过前次120% | 每次训练 |
| 周训练容量 | Σ单次容量 | ACWR 0.8-1.3 | 每周 |
| 局部肌群容量 | 按12肌群分组计算 | 不超过恢复能力 | 每次训练 |
| CNS负荷 | 爆发力+大重量组数占比 | ≤30%总训练量 | 每次训练 |

#### C6.4 动作间衔接逻辑

动作顺序不是随机的，必须遵循运动科学逻辑：

```
正确顺序（不可颠倒）：
1. 爆发力/速度（高翻、冲刺、跳箱）
   ↓ 原因：需要最高神经兴奋度
2. 奥林匹克举重变式
   ↓
3. 复合力量动作（深蹲、硬拉、卧推）
   ↓ 原因：多关节大肌群优先
4. 辅助力量动作（腿屈伸、弯举）
   ↓ 原因：单关节小肌群在后
5. 核心/稳定性训练
   ↓ 原因：核心疲劳不影响前面
6. 纠正性/康复训练
   ↓ 原因：低强度收尾
7. 拉伸放松

反例（Agent必须拒绝生成）：
✗ 先做弯举再做深蹲（小肌群疲劳影响大肌群）
✗ 先做核心再做硬拉（核心疲劳影响脊柱稳定）
✗ 拉伸后做大重量训练（降低力量输出）
```

#### C6.5 热身方案匹配规则

热身必须根据主训练内容匹配，不可通用化：

| 主训练类型 | 热身重点 | 时长 |
|-----------|---------|:--:|
| 下肢力量（深蹲/硬拉） | 髋关节活动度 + 踝背屈 + 核心激活 | 12-15min |
| 上肢力量（卧推/推举） | 肩袖激活 + 胸椎活动度 + 肩胛稳定 | 10-12min |
| 爆发力/跳跃 | 动态拉伸 + 神经激活 + 低强度预演 | 12-15min |
| 有氧/耐力 | 渐进式心率提升 + 动态拉伸 | 8-10min |
| 康复/矫正 | 轻柔活动度 + 目标肌群激活 | 5-8min |

#### C6.6 拉伸方案匹配规则

拉伸必须针对本次训练的目标肌群，不可用通用模板：

| 训练内容 | 拉伸重点 | 每肌群时长 |
|---------|---------|:--:|
| 深蹲/下肢 | 股四头、腘绳、髋屈、臀大肌 | 30s×2 |
| 卧推/上肢 | 胸大肌、三角肌前束、肱三头 | 30s×2 |
| 硬拉/后链 | 腘绳、竖脊肌、臀大肌 | 30s×2 |
| 爆发力/全身 | 本次主要发力肌群 + 拮抗肌群 | 30s×2 |

#### C6.7 质量检查清单

每个计划输出前，必须通过以下检查：

```
□ 是否包含完整的热身→训练→拉伸流程？
□ 动作顺序是否符合运动科学逻辑？
□ 训练容量是否在安全阈值内？
□ 热身是否匹配主训练内容？
□ 拉伸是否针对本次训练目标肌群？
□ 晨间/午间/晚间方案是否覆盖？
□ 动作间衔接是否有逻辑关系（非随机组合）？
□ 是否经过SAFETY-OFFICER审查？
□ 是否经过REHAB-SPECIALIST审查（如有伤病史）？
□ 英语/读书计划是否有相应的专业Agent审查？
```

> **如果任何一项未通过，计划不得输出给用户。**

| 属性 | 内容 |
|------|------|

**容量评估流程**：

```
训练前 → 采集恢复数据（睡眠、HRV、酸痛度）
         → 自动调整当日基准容量（±15%范围内）
训练中 → 实时RPE录入、动作完成度记录
训练后 → 综合判断：
         刚好 ✅ → 维持容量
         过量 ⚠️ → 自动降载（下次-10~20%），连续2次触发预警
         不足 📈 → 自动增量（下次+5~10%），连续3次建议重新评估
```

**安全边界**：
- 单次上调 ≤ 15%
- 单次下调 ≤ 20%
- ACWR 不得超过 1.3
- 连续过量 → 强制插入减载周

#### D8. Agent反馈闭环 ⭐

训练数据不是一次性消费，而是持续回流到Agent体系：

```
训练执行 → 结构化执行报告
                ↓
          每日汇总：
            FIT-COACH：容量是否合适？动作选择是否匹配？
            SAFETY-OFFICER：有无疼痛标记？RPE是否异常？
            REHAB-SPECIALIST：旧伤部位有无不适？
            SLEEP-EXPERT：恢复质量与训练强度匹配度？
                ↓
          每周趋势分析：
            识别容量递增模式、动作掌握进度、恢复趋势
                ↓
          下次计划生成时：
            容量模型更新、检查点优先级调整、
            动作推荐优化、强度曲线修正
```

**闭环驱动的持续优化**：
- 用户频繁搜索某动作 → 该动作Tier 1浮层自动增大
- 用户经常跳过的动作 → Agent评估是否替换为更简单的变式
- 连续3次RPE偏高 → 自动降低该动作初始强度
- 某动作完成负荷持续提升 → 可以进阶变式
- 用户从未搜索过某动作 → 降低该动作的检查点提示频率

## 第五部分：交叉融合引擎详细设计

### 5.1 引擎定位

交叉融合引擎（Fusion Engine）是 PrimeAtlas 的核心差异化模块。它不是一个独立功能，而是一个**编排层**，贯穿于目标推导、计划生成、内容分发、素材回流的全链路。

### 5.2 内容交叉规则

#### 规则一：单向内容注入

```
用户标签关联规则：
  IF 用户有标签[语言:英语] AND 用户有标签[运动:篮球]
  THEN 英语学习素材池 += 篮球术语库、NBA战术解说、篮球英文文献

  IF 用户有标签[语言:英语] AND 用户有标签[运动:羽毛球]
  THEN 英语学习素材池 += 羽毛球英文术语、BWF赛事英文解说、羽毛球英文训练资料
```

**原则**：运动/专业内容注入语言学习素材，而非反向。语言服务于用户关心的内容。

#### 规则二：知识注入 ⭐ 读书→训练的评估决策系统

用户读书后问"能否把这些计划加入我的训练？"——这不是简单的是/否，而是一套完整的**多Agent专业评估决策系统**。

##### 评估决策流程

```
用户提问："《十二周像乔丹一样飞》里的训练计划能加入我的计划吗？"
                    ↓
┌─────────────────────────────────────────────────────┐
│ 阶段一：知识提取                                      │
│ 系统提取书中所有可执行的训练建议，结构化解析            │
│ 每条建议：训练类型、频次、时长、强度、器材、前置条件      │
└────────────────────────┬────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────┐
│ 阶段二：用户画像加载                                   │
│ 加载完整画像：身体现状、伤病史、恢复状态、可用时间、       │
│ 环境设备、当前计划、ACWR、FMS分数等                     │
└────────────────────────┬────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────┐
│ 阶段三：多Agent并行评估                                │
│                                                      │
│ 第一轮（并行，阻断级Agent优先）：                       │
│   FIT-COACH：目标关联度、身体匹配、环境可行性、优先级     │
│   SAFETY-OFFICER：安全性、ACWR、运动负荷（阻断级）       │

│   TIME-ORCHESTRATOR：时间可行性、与现有计划兼容          │
```

##### 评估维度矩阵

对书中每条训练建议，从9个维度综合评估：

| 维度 | 评估内容 | 负责Agent | 权重 |
|------|---------|-----------|:--:|
| 目标关联度 | 是否服务于用户当前目标？ | FIT-COACH | 30% |
| 身体现状匹配 | 用户当前体能能否执行？ | FIT-COACH | 15% |
| 安全性 | ACWR是否超阈值？有受伤风险吗？ | SAFETY-OFFICER | 阻断级 |
| 康复风险 | 是否触发损伤史风险？ | REHAB-SPECIALIST | 阻断级 |
| 时间可行性 | 可用时间能否容纳？ | TIME-ORCHESTRATOR | 10% |
| 环境可行性 | 场地/设备是否支持？ | FIT-COACH | 10% |
| 恢复状态 | 当前恢复状态是否允许？ | SLEEP-EXPERT | 10% |
| 计划兼容 | 能否融入现有计划不冲突？ | TIME-ORCHESTRATOR | 15% |
| 优先级 | 相比现有计划优先级如何？ | FIT-COACH | 10% |

> 阻断级：SAFETY-OFFICER 或 REHAB-SPECIALIST 判定风险 → 强制拒绝，不进入后续评估

##### 决策示例

```
用户：《十二周像乔丹一样飞》第3周"每周3次最大弹跳训练"

✅ 建议1：落地缓冲训练
  → 决策：采纳。与弹跳目标直接相关，无安全风险，可直接融入。

🔄 建议2：每周3次最大弹跳训练
  → 决策：改造采纳。
  理由：当前FMS分数15，完全最大努力落地冲击风险较高。
  改造：最大努力→85%努力，每次增加5min落地缓冲训练。
  融入：替换周三训练中的跳箱部分。

⏸️ 建议3：连续跳跃障碍物训练
  → 决策：暂缓。
  理由：需要专用障碍物器材，当前无。且左踝旧伤需先通过稳定性评估。
  触发条件：踝关节稳定性测试通过 且 获得障碍物器材后自动评估。

❌ 建议4：每日两次高强度跳跃训练
  → 决策：不采纳。
  理由：①ACWR将超1.5，安全风险高；②与现有力量训练冲突；
  ③恢复状态不支持每日高频高强度。
```

##### 融入机制

采纳的方案不是一股脑塞进去，而是**三阶段渐进融入**：

```
适应期（第1周）：降低强度至70%，作为辅助训练
  → 监控：RPE、酸痛度、ACWR变化
过渡期（第2周）：恢复至85%强度，部分替换原有训练
  → 监控：运动表现变化、恢复质量
稳定期（第3周+）：完全融入，达到书中建议水平
  → 监控：长期趋势、PR触发
```

**原则**：阅读不是独立行为，阅读中的知识应流动到其他领域的实践中。但流动必须经过专业评估，每个决策必须有明确理由。

#### 规则三：素材回流

```
素材回流规则：
  IF 用户完成[训练任务] AND 训练内容包含[技术动作/战术/术语]
  THEN 生成[对应英语学习卡片]
  示例：完成"挡拆战术训练" → 系统生成"Pick and Roll 英语学习卡片"
         包含：术语拼写、发音、战术描述英文、相关表达

  IF 用户完成[训练任务] AND 用户有标签[教练目标]
  THEN 生成[教练英语练习素材]
  示例：完成"体能训练" → 系统生成"如何用英语指导体能训练"练习素材
```

**原则**：用户的真实经历是最好的学习素材。训练内容自动转化为学习内容。

#### 规则四：内容主题对齐

```
主题对齐规则：
  IF 用户有多个领域标签
  THEN 所有领域的学习/训练内容优先选择主题重叠的内容
  示例：篮球教练用户
    - 英语听力素材：NBA教练采访 > 通用英语新闻
    - 阅读推荐：运动训练学 > 通用畅销书
    - 写作练习：训练计划英文版 > 通用英文写作
```

### 5.3 统一时间编排算法

#### 编排原则

1. **不同时原则**：同一时间段只安排一个领域的任务
2. **能量匹配原则**：高能量时段 → 高强度任务，低能量时段 → 低强度任务
3. **间隔原则**：不同领域任务间有缓冲
4. **优先级原则**：目标优先级决定时间分配

#### 编排算法伪代码

```
function dailySchedule(userProfile, goals, energyCurve, timeSlots):
    // Step 1: 确定今日各领域时间预算
    timeBudget = allocateTime(goals, timeSlots)
    
    // Step 2: 匹配能量曲线
    for each task in taskPool:
        task.slot = matchEnergy(task.intensity, energyCurve, timeSlots)
    
    // Step 3: 交叉内容注入（K规则）
    for each task in schedule:
        task.content = injectCrossContent(task, userProfile.labels)
    
    // Step 4: 冲突检测
    schedule = resolveConflicts(schedule)
    
    // Step 5: 素材回流标记
    for each task in schedule:
        if task.type == TRAINING:
            schedule += generateLearningCard(task, userProfile.labels)
    
    return schedule
```

#### 时间分配权重

| 因素 | 权重 | 说明 |
|------|------|------|
| 目标优先级 | 40% | 核心目标获得更多时间 |
| 目标进度 | 25% | 落后目标获得补偿时间 |
| 能量匹配 | 20% | 能量决定任务类型分配 |
| 新鲜度 | 15% | 避免长期同一模式 |

### 5.4 场景自适应规则

#### 场景定义

| 场景 | 触发条件 | 编排策略 |
|------|---------|---------|
| 正常工作日 | 默认 | 标准编排 |
| 加班 | 日历标记/时间检测 | 训练压缩至50%，学习切换为碎片模式 |
| 出差 | 位置变化+日历 | 训练切换为自重/酒店模式，学习保持 |
| 生病/恢复 | 手动标记/健康数据 | 训练暂停，学习降频，增加恢复内容 |
| 假期 | 日历标记 | 训练时间增加，学习保持或降低 |
| 碎片日 | 日历零散 | 全部切换为15分钟短单元 |
| 精力低谷 | 能量评分低 | 全部降为低强度，增加恢复内容 |
| 高能日 | 能量评分高 | 优先安排高强度训练，可突破性训练 |

#### 自适应公式

```
调整后计划 = 原计划 × 场景系数

场景系数 = {
    正常工作日: {训练: 1.0, 学习: 1.0, 阅读: 1.0}
    加班:        {训练: 0.5, 学习: 0.5, 阅读: 0.5}
    出差:        {训练: 0.6, 学习: 1.0, 阅读: 1.0}  // 训练切换为旅行模式
    生病:        {训练: 0.0, 学习: 0.3, 阅读: 0.5}  // 训练暂停
    假期:        {训练: 1.5, 学习: 0.8, 阅读: 1.2}
    碎片日:      {训练: 0.7, 学习: 0.8, 阅读: 0.8}  // 全部短单元
}
```

### 6.1 四种调性定义

| 调性 | 人格描述 | 适用用户 | 一句话概括 |
|------|---------|---------|-----------|
| **热血型** | 燃、炸、充满能量 | 需要激励、喜欢挑战的用户 | "冲！今天离目标又近了一步！" |

### 7.1 能量球交互

#### 交互设计

```
能量球 = 首页中央的视觉核心
状态：
  - 未激活：暗色球体，微弱的呼吸光效
  - 等待激活：球体微微发光，表面有细微的能量流动纹理
  - 激活中（长按）：光效从按压点向外扩散
  - 解锁瞬间：爆炸扩散光效，球体点亮
  - 注：原型 v6 采用「长按解锁」，体验已验证良好（非双指按压）
  - 已激活：明亮球体，内部火焰燃烧，颜色由目标进度决定
```

#### 长按扩散解锁

```
操作流程：
1. 用户长按能量球
2. 从触碰点开始，光效向外扩散
3. 扩散覆盖整个球体表面（约0.8秒）
4. 球体"爆炸"点亮，进入激活状态
5. 火焰显现，展示今日目标进度

设计意图：
- 长按 = 仪式感，不是简单的点击
- 扩散解锁 = 爆炸开始的感觉，情绪唤醒
- 物理触感 = 与手机的物理连接，增强真实感
```

### 7.2 火焰系统

#### 火焰的含义

```
火焰 ≠ 坚持天数
火焰 = 离目标有多近

火焰状态：
  旺盛燃烧：目标进展顺利，超过预期
  稳定燃烧：目标进展正常
  小火苗：  目标进展缓慢，需要关注
  微光：    目标停滞，需要调整
  休息日：  火焰不灭，但切换为柔和的"养护模式"（蓝色/紫色光）
```

#### 火焰视觉参数

| 参数 | 旺盛 | 稳定 | 小火苗 | 微光 | 休息日 |
|------|------|------|--------|------|--------|
| 火焰高度 | 高 | 中 | 低 | 极低 | 无（蓝色光） |
| 火焰颜色 | 蓝白+橙 | 橙+黄 | 黄+红 | 暗红 | 蓝紫渐变 |
| 亮度 | 高 | 中 | 低 | 暗 | 柔和 |
| 动态速度 | 快 | 中 | 慢 | 极慢 | 缓慢呼吸 |
| 粒子效果 | 大量火星 | 少量火星 | 无 | 无 | 星光粒子 |

#### 火焰计算模型

```
目标进度综合分 = Σ(各维度进度 × 维度权重) / Σ维度权重

注（动态权重）：未激活/未使用的维度权重记为 0，分母随用户**实际使用的维度**动态变化——单域用户不会被「缺失维度」惩罚；权重由用户在身份定义时隐式确定（使用的维度即参与计算）。

火焰状态 = f(目标进度综合分, 近期趋势)

  if 综合分 > 110%（超额完成）    → 旺盛
  if 综合分 90%-110%              → 稳定
  if 综合分 60%-90%               → 小火苗
  if 综合分 < 60%                 → 微光
  if 今日标记为休息日              → 养护模式

近期趋势修正：
  if 连续3天上升 → 火焰状态+1（微光→小火苗，小火苗→稳定）
  if 连续3天下降 → 火焰状态-1
```

### 7.3 目标承诺仪式

#### 仪式设计

```
目标承诺仪式 = 每周一次的目标确认和重新承诺

流程：
1. 系统展示本周目标概览（30秒回顾）
2. 展示上周完成情况 + PR亮点
3. 展示下周目标预览
4. 核心仪式：用户按下"承诺"按钮
   - 按钮按下时：能量球爆发出光效
   - 承诺完成：能量球火焰重新燃起
   - 伴随文案（调性适配）

仪式目的：
- 不是机械的"确认"，是有情感投入的承诺
- 建立"我选择这样做"的自主感
- 每周一次的节奏感
```

#### 承诺文案（按调性）

| 调性 | 承诺引导语 | 确认语 |
|------|-----------|--------|
| 热血型 | "这周的目标就在眼前，你敢接吗？" | "接了！这周让它燃烧！" |
| 专业型 | "本周目标已优化，确认开始执行。" | "目标已确认。" |
| 陪伴型 | "新的一周，我们继续一起走。" | "嗯，一起走。" |
| 严厉型 | "目标在这里，确认就开始。" | "确认。" |

### 7.4 休息日仪式

#### 休息日设计

```
休息日规则：
- 休息日是计划的一部分，不是"失败"
- 休息日能量球火焰不灭（切换到养护模式）
- 休息日不推送任务提醒
- 休息日可以查看成长数据和回顾
- 休息日结束后，火焰自动恢复

休息日文案（按调性）：
  热血型： "今天的休息，是为了明天更猛烈的燃烧！"
  专业型： "计划内休息日。身体恢复中，预计明天恢复率98%。"
  陪伴型： "今天是休息日，好好照顾自己。"
  严厉型： "休息日。好好恢复，明天继续。"
```

### 7.5 PR庆祝

#### PR类型与触发

| PR类型 | 检测条件 | 庆祝强度 | 示例 |
|--------|---------|---------|------|
| **突破PR** | 单一指标突破历史最佳 | 高 | 卧推1RM突破、英语测试分数新高 |
| **坚持PR** | 长期稳定执行（如连续4周完成率>85%） | 中 | "你已经稳定训练4周了" |
| **觉醒PR** | 认知/方法层面的突破 | 中 | "你发现了间歇训练比持续训练更有效" |
| **里程碑PR** | 达到阶段性目标 | 极高 | 完成教练资格证、完成第一场回归赛 |

#### PR庆祝体验流程

```
1. 触发检测（系统自动）
2. 屏幕出现特殊光效（不同于普通通知）
3. 用户点击 → 全屏庆祝动画
4. 成果卡片展示（具体数据+身份关联）
5. 可选操作：
   - 保存到PR收藏
   - 分享（生成分享图）
   - 查看详情
   - 关闭（不分享，仅保存）
6. 默认行为：保存到PR收藏，不公开分享
```

#### PR分享卡片设计

```
分享卡片包含：
- PR类型图标
- 核心数据（如"+15% 卧推1RM"）
- 身份关联（如"离国际篮球教练又近了一步"）
- 日期
- 调性适配的视觉风格
- 可选：一句话感想（用户自定义）

不包含：
- 连续打卡天数
- 排行榜排名
- 任何对比他人的数据
```

## 第九部分：多Agent协作决策体系 ⭐

> **核心设计**：每个决策阶段由专业Agent主导设计，相关Agent交叉评审，达成一致后输出计划。

### 9.1 Agent角色与权限矩阵

#### 权限等级

| 权限 | 标识 | 含义 |
|------|:----:|------|
| **主导权** | L | 在该领域有最终设计权 |
| **否决权** | V | 安全/健康问题可一票否决，必须重新设计 |
| **强建议权** | S | 建议必须被明确回应，拒绝需记录理由 |
| **建议权** | A | 可提出建议，主导Agent可选择采纳 |

#### 12个Agent角色

| Agent | ID | 职责 | 权限 |
|-------|-----|------|:----:|
| **体能训练专家** | FIT-COACH | 主导训练计划设计（动作、强度、周期） | L |
| **安全员** | SAFETY-OFFICER | ACWR监控、疼痛熔断、动作安全审查 | V |
| **营养师** | NUTRITIONIST | 营养建议、热量和宏量营养素评估 | S |
| **睡眠质量专家** | SLEEP-EXPERT | 恢复质量评估、睡眠建议 | S |
| **体能康复专家** | REHAB-SPECIALIST | 伤病风险评估、康复方案设计 | V |
| **语言学习专家** | LANG-COACH | 主导英语学习路径设计 | L |
| **语言评测专家** | LANG-ASSESSOR | 难度评估、进度合理性评审 | S |
| **阅读引导专家** | READING-GUIDE | 主导读书计划设计 | L |
| **知识转化专家** | KNOWLEDGE-TRANSFER | 评审行动转化方案可行性 | S |
| **时间编排专家** | TIME-ORCHESTRATOR | 统筹三个方向时间分配 | L |
| **调性适配专家** | TONE-ADAPTER | 确保输出风格匹配用户调性 | S |
| **用户画像Agent** | USER-PROFILE | 维护用户画像，为所有Agent提供上下文 | A |

### 9.2 协作流程

#### 场景A：创建新目标 → 生成初始计划

```
用户输入目标 → USER-PROFILE 提供画像上下文
                    ↓
    ┌───────────────┼───────────────┐
    ↓               ↓               ↓
FIT-COACH      LANG-COACH     READING-GUIDE
(主导训练设计)  (主导学习设计)   (主导阅读设计)
    ↓               ↓               ↓
SAFETY-OFFICER  LANG-ASSESSOR  KNOWLEDGE-TRANSFER
(安全评审)      (难度评审)      (转化评审)
    ↓
REHAB-SPECIALIST (康复风险评审)
    ↓
NUTRITIONIST + SLEEP-EXPERT (营养+睡眠建议)
    ↓
TIME-ORCHESTRATOR (统一时间编排)
    ↓
TONE-ADAPTER (调性适配输出)
    ↓
用户收到计划（可查看Agent协作过程）
```

**协作规则**：
- 主导Agent并行设计（节省时间）
- 安全Agent串行评审（安全优先）
- 专业Agent并行评审
- 有否决权Agent未通过 → 退回主导Agent重新设计
- 全过程约60-90秒

#### 场景B：日常调整（加班/出差）

```
环境变化触发 → TIME-ORCHESTRATOR 重新计算时间分配
                    ↓
    FIT-COACH / LANG-COACH / READING-GUIDE 并行调整
                    ↓
    安全Agent快速审查（仅审查变更部分）
                    ↓
    TONE-ADAPTER 输出调整通知
```

轻量快速，≤20秒。时间缺口≤30分钟时跳过评审。

#### 场景C：周度复盘与迭代

```
数据收集 → 各Agent并行分析上周数据
                ↓
    各Agent提出迭代建议
                ↓
    TIME-ORCHESTRATOR 检测冲突
                ↓
    有冲突 → 仲裁流程
    无冲突 → 合并输出
                ↓
    TONE-ADAPTER 生成周报
```

#### 场景D：安全事件触发

```
疼痛标记 / 异常数据 → SAFETY-OFFICER 立即介入
                            ↓
    熔断判定 → 立即暂停相关训练
                            ↓
    REHAB-SPECIALIST 评估康复方案
                            ↓
    FIT-COACH 根据康复方案调整训练计划
                            ↓
    TIME-ORCHESTRATOR 重新编排时间
```

安全Agent走独立快速通道，不等待其他Agent。

### 9.3 评审与仲裁机制

#### 四级仲裁递进

```
Level 1：主导Agent回应
  "你的建议我接受/不接受，理由是..."

Level 2：多Agent讨论
  双方各陈理由，其他相关Agent投票

Level 3：权威投票
  按权重计票：
  SAFETY-OFFICER / REHAB-SPECIALIST = 3票（安全优先）
  其他Agent = 1票
  平票时安全方胜出

Level 4：用户决策
  "专家团队存在分歧，请你决定："
  展示双方理由，用户投票
```

#### 安全否决独立通道

安全Agent的否决不进入常规仲裁：
- `SAFETY-OFFICER` 否决 → 无条件生效
- `REHAB-SPECIALIST` 否决 → 无条件生效
- 立即通知用户，附带详细理由和替代方案

### 9.4 用户可见的协作呈现

用户可以看到Agent在协作，而非黑盒输出：

```
默认视图（结论）：
  "经过体能训练专家设计、安全员审查、营养师建议，
   你的训练计划已生成。"

展开视图（对话记录）：
  FIT-COACH: "基于你的弹跳目标，建议4×5深蹲，每周递增2.5kg"
  SAFETY-OFFICER: "ACWR检查通过。但第4周递增后接近阈值，建议插入减载周。"
  FIT-COACH: "同意。在第4周后加入减载周。"
  NUTRITIONIST: "训练强度需要额外200kcal/天，建议增加蛋白质摄入。"
  → 最终计划已融合3条建议

深度视图（原始日志）：
  完整的Agent推理链，供高级用户和调试使用
```

**协作状态指示器**：
- 🔵 协作中：Agent正在讨论
- 🟢 已达成一致：所有Agent通过
- 🟡 有分歧：展示分歧点和仲裁状态
- 🔴 安全否决：安全Agent已介入

## 第十部分：技术架构概要

### 10.1 架构原则

1. **离线优先**：核心功能（今日视图、执行追踪、能量标记）离线可用
2. **事件驱动**：交叉融合引擎基于事件触发，非轮询
3. **Agent协作**：Agent间事件驱动+消息队列混合通信
4. **规则引擎**：交叉融合规则可配置，支持热更新
5. **调性解耦**：调性系统与业务逻辑完全解耦，通过适配层注入
6. **隐私优先**：所有用户数据默认端到端加密

### 9.3 大模型架构 ⭐

#### 9.3.1 设计原则

> **模型可配置、可切换，不绑定任何单一供应商。**

- 当前主模型：**DeepSeek V4**（或最新版本）
- 架构预留切换能力：GPT-5.5、Claude Fable系列、GLM-5.2、DeepSeek系列等
- 不同Agent可使用不同模型（安全Agent可用更保守的模型，创意Agent可用更灵活的模型）
- 用户可在设置中查看和切换模型偏好

#### 9.3.2 模型路由层架构

```
┌─────────────────────────────────────────────────────┐
│                  Model Router (模型路由层)            │
│                                                      │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐           │
│  │ DeepSeek │  │  GPT-5.5 │  │  Claude   │  ...     │
│  │   V4     │  │          │  │  Fable    │           │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘           │
│       │              │              │                 │
│  ┌────┴──────────────┴──────────────┴────┐           │
│  │         Unified Adapter Layer          │           │
│  │   - 统一请求格式                       │           │
│  │   - 统一响应解析                       │           │
│  │   - Token计费统一                      │           │
│  │   - 速率限制管理                       │           │
│  │   - 故障自动切换(Fallback)             │           │
│  └────────────────┬──────────────────────┘           │
│                   │                                  │
│  ┌────────────────┴──────────────────────┐           │
│  │         Agent Model Assignment         │           │
│  │   FIT-COACH      → DeepSeek V4 (主)    │           │
│  │   SAFETY-OFFICER → DeepSeek V4 (主)    │           │

│  │   NUTRITIONIST   → DeepSeek V4 (主)    │           │
│  │   ...可配置...                         │           │
│  └────────────────────────────────────────┘           │
└─────────────────────────────────────────────────────┘
```

#### 9.3.3 模型配置能力

**系统级配置**（管理员）：
```yaml
model_registry:
  deepseek:
    provider: DeepSeek
    model_id: deepseek-v4
    endpoint: https://api.deepseek.com/v1
    capabilities: [text, reasoning, code]
    default_for: [FIT-COACH, SAFETY-OFFICER, LANG-COACH, TONE-ADAPTER, ...]
    
  gpt55:
    provider: OpenAI
    model_id: gpt-5.5
    endpoint: https://api.openai.com/v1
    capabilities: [text, reasoning, vision, code]
    available_as: alternative
    
  claude_fable:
    provider: Anthropic
    model_id: claude-fable-latest
    endpoint: https://api.anthropic.com/v1
    capabilities: [text, reasoning, vision]
    available_as: alternative
    
  glm52:
    provider: ZhipuAI
    model_id: glm-5.2
    endpoint: https://open.bigmodel.cn/api/paas/v4
    capabilities: [text, reasoning, code]
    available_as: alternative
```

**用户级配置**：
- 在"我的→设置→AI模型"中查看当前使用的模型
- 可切换主模型（从可用模型列表中选择）
- 切换后所有Agent统一使用新模型
- 高级用户可为不同Agent分配不同模型（可选）

**Agent级配置**：
- 每个Agent可独立指定模型
- 安全Agent：优先稳定性和安全性，可选更保守的模型
- 创意Agent（TONE-ADAPTER）：优先语言表达能力
- 推理Agent（FIT-COACH）：优先逻辑推理能力

#### 9.3.4 Fallback机制

```
主模型（DeepSeek V4）
    │
    ├── 正常 → 使用主模型
    │
    ├── 超时（>30s） → 自动切换备用模型
    │
    ├── 错误率 > 5% → 自动切换备用模型
    │
    └── 完全不可用 → 降级到本地规则引擎（核心功能不中断）

备用模型优先级：GPT-5.5 → Claude Fable → GLM-5.2
```

#### 9.3.5 模型使用策略

| Agent类型 | 调用频率 | 延迟要求 | 推荐模型特性 |
|-----------|:------:|:------:|------------|
| 计划生成Agent | 低（日/周） | ≤90s | 强推理能力 |
| 安全审查Agent | 中（每次训练） | ≤30s | 稳定可靠 |
| 实时调整Agent | 高（事件触发） | ≤20s | 快速响应 |
| 调性适配Agent | 高（每次输出） | ≤5s | 语言表达 |
| 知识提取Agent | 低（按需） | ≤60s | 深度理解 |
| 对话Agent | 中（用户触发） | ≤10s | 自然对话 |

### 9.4 关键技术选型建议

| 组件 | 建议方案 | 理由 |
|------|---------|------|
| 移动端框架 | SwiftUI + Jetpack Compose | 原生性能，支持复杂动画（Prime Pulse） |
| 后端框架 | Go / Rust | 编排算法需要高性能 |
| **AI模型主模型** | **DeepSeek V4** | 当前主模型，性价比高，推理能力强 |
| AI模型备选 | GPT-5.5 / Claude Fable / GLM-5.2 | 可配置切换 |
| 模型路由 | 自研Model Router | 统一适配、自动Fallback |
| 规则引擎 | 自研DSL或Drools | 交叉融合规则需要灵活配置 |
| 实时通信 | WebSocket | PR推送、实时进度更新 |
| 数据同步 | CRDT | 离线优先的多端同步 |
| 动画引擎 | Lottie + 自定义Metal/OpenGL | 火焰效果需要高性能渲染 |
| 数据库 | PostgreSQL + TimescaleDB | 时序数据（训练记录）+ 关系数据 |

|------|------|

| 能量球交互响应 | < 100ms |
| 离线可用性 | 核心功能100%离线可用 |
| 数据同步延迟 | < 5秒（正常网络） |
| 推送到达率 | > 95% |
| 崩溃率 | < 0.1% |
| 数据安全 | E2E加密，GDPR合规 |

## 第十一部分：Sprint 建议

### 10.1 总体原则

- **不做MVP裁剪**：所有功能最终都要交付
- **按敏捷Sprint排优先级**：先交付核心差异化价值，再补全周边
- **每个Sprint交付可体验的完整功能**：不是"做完一半的功能"，而是"完整可用的子集"
- **调性系统随功能渗透**：不单独做一个"调性Sprint"，而是在每个功能中植入调性适配
